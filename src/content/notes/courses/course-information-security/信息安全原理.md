# 一、信息安全简介
## 1. 信息安全的概念与基础
1. 永远**不可能**有**完美**的安全系统
	- 异地备份数据
	- 技术手段防止信息泄露
	- 断开互联网防止篡改
2. **安全的衡量**
	- 安全核心：privacy 隐私、availability 可用性、dependability 可靠性
	- 现实考量：speed 速度、money 成本、new features 新功能、profitability 盈利能力
	- **需做成本收益权衡**
	- 例子：摄像头、福特汽车配件、保险
3. **最常见的安全误区**：摘自《安全计算》杂志
	1. 我的家用电脑/笔记本电脑不会受到黑客的攻击
		- **黑客攻击的目的**：刺激，**金钱**
	2. 我们公司是不会被黑客攻击的，黑客不会对我们这样的公司下手的
	3. 内部网络中的服务器不会受到攻击
	4. 如果我安装了防火墙，我的网络就不会被黑客入侵了
	5. 我私人网络中的人员都是可以信赖的；**大多数安全漏洞都是由公司外部人员造成的**
		- 80%的攻击来自内部人员
		- **内部人员** Insider **定义**：**合法访问或关联**系统/环境某部分的人，相比外部入侵者，拥有更多机会和系统知识，通常有明确动机，如报复、经济利益、获取信息等
		- **内部人员范围扩大**：受扁平化世界 Flat World 影响，企业资源分布式发展，内部人员**不再局限于公司员工**，还包括开发者、测试人员、销售团队、顾问、物流人员、客户及客户内部人员等，内外部、盟友与敌人的边界模糊
	6. **黑客只是想证明自己能入侵网络的极客**
		- 并非仅为刺激、挑战，核心是获取利益，包括金钱、战略优势、政治变革、商业情报等
4. **数据主权** Data Sovereignty
	- **定义**：数据主权是指**数据受制**于其收集的**国家**内的法律和治理结构
	- 例子：伊朗信息设施全部被美控制，仅中国有自己独立系统；特斯拉不准进入政府机关
5. 网络攻击的**报道现状**：**公开报道的网络攻击事件远少于实际发生的数量**
	- 所谓15岁少年成为超级黑客是假象，实际是专业黑客的掩护
	- 企业因消费者信心受损的经济损失，远大于黑客攻击本身，因此不到一成的严重网络入侵事件会向官方报告
## 2. 信息安全的历史
1. 信息安全历史：
	1. 历史意义：**加密** Encryption，如何保护需要传递的信息
	2. 中国古代密码
		1. 中国历史上最早的**密码** cipher：**太公兵法**
			- 信息替代 substitution；信息位移 displacement
			- 是方法而不是严格上的密码
		2. 中国最早的**密码通信** **CCP**：**豪密** 周恩来编写
	3. 外国古代密码
		1. **斐斯托斯圆盘** Phaistos Disc：烧制黏土圆盘
			- 作为早期信息载体，其未知文字与符号体现了人类对信息编码与保密的早期探索
			- **至今未被完全破译**，印证了密码学中**样本量不足则难以破译**的原则
		2. **凯撒密码** Caesar cipher
			1. 已知**最早的替换密码** Substitution Cipher
			2. 加密规则：将明文中的每个字母按字母表顺序向后或向前移动固定位数
			3. **密钥空间**：对于26个英文字母，共有25种可能的位移
				- 密钥即为**位移的位数**
		3. **棍子密码** Scytale Cipher
			1. **加密原理**
				将长条状皮革螺旋缠绕在木棒上，沿木棒水平方向书写明文，解密方**使用相同直径的木棒**，将密文纸条重新螺旋缠绕在木棒上，文字会自动在水平方向对齐，还原出可读明文
			2. 属于**置换密码**
			3. **密钥**：**木棒的直径**，决定缠绕圈数和列数
		4. **隐写术** Steganography
			1. **定义与本质**
				- 隐写术是将秘密信息隐藏在其他公开载体中，使第三方**无法察觉信息存在**的技术
				- **与密码学不同**：密码学是让信息**不可读**，隐写术是让信息**不可见**
			2. 古代案例
				- **奴隶头皮传信**：在头皮上面写下密信，待头发长出后传递
				- **蜡板藏信**：将信息写在蜡板的木质基底上，再用蜡覆盖，使蜡板看起来一片空白
			3. 现代案例
				- **蛋中写字**：用明矾和醋制成的墨水在鸡蛋上书写，然后煮熟鸡蛋。当蛋壳被剥去后，蛋清上的字迹依然清晰可见
				- **微点** Microdot：将信息缩印在极小的胶片上。接收方需要特殊的投影设备才能读取信息
2. 密码学 Cryptography 的**演进**
	1. **目标**
		- 密码学的主要目标是**保密**
		- 历史上的保密方式是人类智慧的精华，但密码学真正走向科学化，要归功于**柯克霍夫原则**及其后续发展
	2. 四次关键演进
		1. 第一次演进：**柯克霍夫原则** Kerckhoffs' Principle
			- 原文：一个密码系统应当是安全的，即使**除了密钥之外，关于该系统的一切都是公开知识**
			- **思想**：假设任何**不变**的东西**都会被公开**
			- **意义**：
				- 使密码设计从**史前时期走向经典时期**
				- 奠定了密码**工程化、规模化应用的基础**
				- 构建了现代密码安全的**核心评估准则**
				- 为密码系统的迭代与加固提供了**底层逻辑**
		2. 第二次演进：**计算机** Computers
			- **出现动因**：二战期间为破解德国恩尼格玛 ENIGMA 密码机，催生了早期电子计算机的诞生
			- **理论基础**：**香农信息论**
			- 计算机的出现极大**提升了密码算法的复杂度和运算速度**，催生了如DES、AES等现代对称加密标准
			- 80年代个人PC MSDos 90年代视窗
		3. 第三次演进：**公钥密码** Public key ciphers
			- **代表**：Diffie-Hellman密钥交换、RSA算法等
			- **意义**：
				- 解决了对称密码中的**密钥分发难题**，发送方和接收方无需预先共享任何密钥，就可以安全地交换大量秘密信息
				- 实现了**数字签名和身份认证**，为现代互联网安全，如HTTPS、电子支付奠定了基础
		4. 第四次演进：**互联网/移动互联网** The Internet/Mobi-Internet
			1. **演进变化**：密码学不仅是保密的工具，而是扩展到**解决认证、完整性、不可否认性等更多安全问题**
			2. 应用领域的扩展
				- 从传统的**军事**领域，逐步拓展到**民用**领域，再深入到**个人**场景
				- **保护方式的升级**：从简单的点对点协议PTP，发展到复杂的PTP协议，再到各种互联网协议及应用，如5G、区块链、物联网IoT和AI
			3. 软硬件发展带来的挑战与机遇
				- 硬件性能和容量的提升、成本的降低，为密码学提供了**更强大的算力支持**
				- 催生了大规模分布式计算、云计算、密码分析等新领域，同时也对密码算法的安全性和效率提出了更高要求
3. 计算机安全的现代发展和特性
	1. 时代发展特征
		- **PC时代**：病毒肆虐，目的一般为**炫技/破坏**，如黑色星期五、CIH 硬件破坏病毒、办公宏病毒
		- **互联网时代**：黑客/蠕虫/DOS爆发，目的一般为**牟利**，如熊猫烧香、中美黑客战、nimda蠕虫病毒、冲击波 Blaster/震荡波Sasser 蠕虫、僵尸网络 botnet、CSDN密码泄露
		- **后（移动）互联网时代**：**隐私/国家数据安全为主**，如棱镜门、WPA2/Krack漏洞、华为/TikTok事件，ios7越狱太极门
	2. 计算机安全与传统**纸质信息安全**的区别
		1. 数字信息的特性导致安全难度提升：
			- 数字**副本与原件无差别**，纸质**可区分**
			- 数字**篡改无痕迹**，纸质篡改留**物理印记**
			- 数字信息**易删除销毁**，纸质相对**稳定**
			- 数字签名依赖**二进制**，纸质签名/印章**依赖物理特征**
		2. 纸质信息安全
			- 骑缝章盖标书
			- 空白纸签名
	3. 计算机安全的五大固有特性
		- **综合性**：系统安全取决于最薄弱环节
		- **过程性**：安全是持续博弈、螺旋上升的模型
		- **动态性**：安全系统需持续更新优化
		- **层次性**：需多层安全技术/方法解决风险
		- **相对性**：无绝对的安全，安全是相对状态
4. 计算机安全基础概念
	1. 信息安全的**三大基石**
		1. **保密性** Confidentiality：确保数据仅能被授权方访问读取，防止未授权的信息泄露；对应**截获类被动攻击**，如网络嗅探、数据窃听
		2. **完整性** Integrity：确保数据不被未授权方插入、删除、篡改，保证数据的真实与完整；对应**篡改类主动攻击**，如报文修改、数据库非法篡改
		3. **可用性** Availability：确保系统、资源和服务可被授权方正常访问，不会被异常中断；对应**中断类主动攻击**，如 DoS/DDoS 拒绝服务攻击
		4. **真实性**/**不可否认性** authenticity / non-repudiation：
			- **真实性**
				- **定义**：确保一条消息、一笔交易或其他信息交换，确实来自它所声称的来源
				- **作用**：涉及身份的证明，即**你是谁**的验证
			- **不可否认性**
				- **定义**：意味着一方有履行合同义务的意图。同时，交易的一方不能否认自己收到了交易，另一方也不能否认自己发送了交易
				- **目的**：**防止交易双方事后抵赖**，保障交易的法律有效性和可追溯性
	2. 资产、威胁、漏洞、风险
		- **资产** Asset：需**保护**的人、物、信息
		- **威胁** Threat：利用漏洞损害资产的因素，无论有意/无意，是需要**加以防范**的对象
			- 分类：按方法、按意图
		- **漏洞** Vulnerability：安全**防护的薄弱点/缺口**
		- **风险** Risk：**威胁利用漏洞造成资产损失**的可能性
		- **公式**：A+T+V=R 资产+威胁+漏洞=风险
	3. **信息安全攻击的类型**
		1. **被动攻击** Passive Attacks：**难检测**，**无法完全避免**，**尽量预防**
			- **截获** Interception：被动攻击，仅截获，未授权获取信息
		2. **主动攻击** Active Attacks：破坏可用性、完整性、真实性，**易检测**，**难预防**
			- **中断** Interception：破坏硬件/通信、删除程序、DOS攻击，让系统不可用
			- **修改** Modification：篡改数据库记录、修改网络传输数据、黑客入侵改系统
			- **伪造** Fabrication：数据库插假记录、伪造IP包、钓鱼邮件/网站，**冒充合法主体**
				- **重放/中间人攻击** replay
		3. 图例 ![Screenshot_20260309_143418](/blog/obsidian-assets/screenshot-20260309-143418-56e0696de5.jpg)
	4. **信息安全的防御**
		1. **防御目标**
			1. **预防** Prevention：阻止攻击者违反安全策略
			2. **检测** Detection：发现攻击者的违规行为
			3. **恢复** Recovery：制止攻击、修复系统并恢复运行
				- 高级版：被攻击后仍能正常工作
		2. **防御的政策与机制**
			- **政策** policy：定义安全规则，规定允许/禁止行为，需避免政策冲突产生漏洞
			- **机制** mechanisms：执行安全政策的技术/流程手段，如防火墙
		3. **六大安全服务**：认证、访问控制、数据保密性、数据完整性、不可否认性、可用性
		4. **运营与人为关键考量**
			- **运营考量** Operational Issues：成本收益分析（预防/恢复）、风险分析（保护对象/程度）、法律法规与习俗（避免违法/难落地）
			- **人为因素** Human Issues：70%管理+30%技术，人为问题是安全漏洞的主要来源，包括未培训人员、社会工程、内外部威胁
			- **整体防御**：需融合目标、政策、机制/服务、运营、人五大维度，形成完整体系
# 二、密码学
## 1. 密码学的概念与历史
1. **密码学基础** Fundamentals of Cryptography
    1. 词源与分类
        1. **词源**：Cryptography 源自希腊语 kruptos 秘密+ graphia 书写
        2. **分类**：
	        - Cryptography **密码学**：Scrambled **打乱**，让信息**不可读**
		        - Substitution **替换**
				    - Code 替换**单词**
			        - Cipher 替换**字母**
		        - Transposition **置换/位移**
	        - Steganography **隐写术**：Hidden **隐藏**，让信息**不可见**
    2. 术语与加解密公式
        1. **基础术语**：**明文** Plaintext,$P$、**密文** Ciphertext,$C$、**加密** Encryption,$E()$、**解密** Decryption,$D()$、**密钥** Key,$K$
        2. **加解密公式**：$C=E_K(P)$$、$$P=D_K(C)$
        3. Cryptographer 密码学家 Cryptanalyst 密码破译者![Screenshot_20260311_101404](/blog/obsidian-assets/screenshot-20260311-101404-0890396643.jpg)
    3. 两种基础加密算法
        1. **替换算法**：将明文字母按规则**替换为其他字母**，分单字母替换、多字母替换
        2. **置换算法**：仅调整明文字母的排列顺序，**不改变字母本身**，如“信息安全导论”置换成“信全息导论安”
	4. **密码分析**
	    1. 基于攻击者信息的攻击类型
	        1. **唯密文攻击** Ciphertext-only：攻击者**仅获取密文**集合
	        2. **已知明文攻击** Known-plaintext：攻击者**掌握部分**密文及对应明文
	        3. **选择明文/密文攻击** Chosen-plaintext/ciphertext：攻击者可**获取自选**明文**对应的**密文，或自选密文对应的明文，但可能难以反向
	        4. **自适应选择明文攻击** Adaptive chosen-plaintext：攻击者可根据前一次加密信息选择后续明文，即只用选择一次，后面的会根据所选解开
	        5. **相关密钥攻击** Related-key attack：攻击者可获取**不同密钥加密**的密文，且**已知密钥间的关系**
	    2. 密码分析的攻击结果
	        1. **完全破解** Total break：攻击者推导出**秘密密钥**
	        2. **全局推导** Global deduction：找到与加解密功能**等效的算法**，无需获知密钥
	        3. **局部（实例）推导** Instance(local) deduction：发现此前未知的明文或密文
	        4. **信息推导** Information deduction：获取关于明/密文的**香农信息**(即有用信息)
	        5. **区分算法** Distinguishing algorithm：能将该**密码与随机置换进行区分**
2. **密码学发展的三个时代**
    1. **划分依据**：**技术实现方式**，分为古典密码时代、机器密码时代、计算机密码时代
    2. **古典密码时代**：手工密码，一战前
        1. 早期经典手工密码：腰带/木棍密码、古希腊密码、凯撒密码
            - 玛丽女王密码：含23个字母符号、35个单词符号，进行替换，1586年被破解，导致巴宾顿阴谋败露
        2. 单字母替换密码的破解：**频率分析**，基于英语中**频率最高的字母是e**
            1. 提出者：9世纪阿拉伯学者阿尔·肯迪，其成果1987年才被重新发现
            2. **思路**：基于自然语言的字母频率特征破解，对特殊文本失效，如不含高频字母e的小说
        3. 抗频率分析的密码：**维吉尼亚密码** Vigenere
            1. 发明者：布莱斯·德·维吉尼亚，1586年发表《密码论》，发明**维吉尼亚方阵**![Pasted image 20260311103703](/blog/obsidian-assets/pasted-image-20260311103703-b357e4440b.png)
            2. 核心：**多字母替换**，通过偏移密钥使同一明文字母对应不同密文字母，**让频率分析失效**，曾300年被认为不可破
        4. 维吉尼亚密码的破解：**卡西斯基测试**
            1. **发现者**：查尔斯·巴贝奇，1854年发现，因英政府要求未发表；弗里德里希·卡西斯基1863年独立发现并公开，该方法因此得名
            2. 破解步骤：
	            1. 利用**密钥重复导致密文重复**，找重复序列间距的因数**确定可能的密钥长度**
	            2. 按密钥长度**拆分密文**，得到多个单字母替换子密文
	            3. **对各子密文做频率分析**，破解每个子密码后**整合**得到完整密钥
        5. **书本密码**：比尔文件为经典案例
            1. 核心：**以某本书为密钥**，将数字对应书籍中单词的**首字母解密**
            2. 比尔文件：1822年诞生，以《独立宣言》为密钥破解第一部分藏宝地址，后两部分至今未解，真实性存疑
            3. 特点：无需特殊训练，普通书籍即可作为密钥，广泛应用于战争、影视，安全性依赖密钥书籍的保密性
    3. **机器密码时代**：机械/电子密码机，一战出现/二战普及
        1. 核心代表：**转子机** Rotor Machine，密码学计算机时代前的最高成就，以德军**恩尼格玛 Enigma 机**为代表
        2. 恩尼格玛机原理
            1. 发明者：亚瑟·舍比乌斯 1919年，德军装备3/4/5转子版本，3转子密钥空间约$1.0×10^{16}$种，5转子约$2.6×10^{26}$种
            2. 核心构成：灯板、键盘、插板、3-5个转子、反射器
            3. **原理**：输入原文后，**每次**按动按键，最右边转子转一格，当转子转一圈时左边转子转一格，来**改变对应的映射**，机器会自动在对应的映射上亮灯。解密时需要知道转子的选择，起始位置，再将密文输入，反向进行解密，军用版增加插板，可初始交换6对字母
            4. 密码机的**密钥**：
	            1. 从五个转子中随意**挑选出三个**，并以自由的**顺序安装**在密码机上
	            2. 三个转子中的每一个都有26种不同的**起始位置**选择
        3. **符合柯克霍夫原则**：
	        1. 算法为机器的物理构造，即整台机器和转子，难以改变
	        2. 密钥为**转子初始设置**的顺序+位置和**插板配置**，为可变/保密
        4. **恩尼格玛机的破解**
            1. 阶段1：波兰 马里安·雷耶夫斯基
	            1. 利用间谍，获取德方恩尼格玛使用手册
	            2. 利用德方**使用失误**，波兰通过数学分析密文的链路长度，与转子配置和插板无关，将密钥搜索范围从1000亿级降至105456种
	            3. 正常将**每日密钥** Day Key 加密为6位，再用**每日密钥加密消息** Message，再将每日密钥结合在消息前并发送，但德方错误不再改变转子顺序，**只将初始位置改变**，并将这3位**重复两遍**，因此第1，4、2，5、3，6位对应的密文有相对关系
	            4. 制作**bomba**炸弹机，20分钟-2小时解出每日密钥，破解超9000条德军信息
            2. 阶段2：英国 艾伦·图灵，布莱切利园
	            1. 德方1939年升级恩尼格玛，增加转子/插板，波兰技术失效
	            2. 利用德军消息的**固定表达/已知明文猜测**，如德语气象、军事固定用语
	            3. 改进bomba机，忽略插板简化复杂度，5小时遍历所有转子组合，彻底破解恩尼格玛
        5. 关键人物：舍比乌斯发明、雷耶夫斯基初步破解、图灵彻底破解
        6. 破解后续：图灵因同性恋受英国当局迫害，1954年服毒自杀；雷耶夫斯基的贡献被丘吉尔评价无数人亏欠少数人太多
    4. 计算机密码时代：二战后
        - 核心特征：二战后诞生，通过**计算机程序实现加密算法**，取代机械/电子密码机，是当前密码学的主流实现方式
2. **密码学启示**
    1. **恩尼格玛机被破解的原因**：**德方的使用误用**，重复加密导致模式暴露；**插板**虽大幅扩大密钥空间，但**容易受到多种攻击**，**转子**配置在**算法**层面更加**安全**，但**会被暴力破解破坏**
    2. 破解加密算法的**通用思路**：**寻找**密文/使用中的**模式**来**降低**破解的**复杂度/维度**；必要时**暴力破解**；依赖数学基础和运气
    3. **柯克霍夫原则的价值**：密码系统的安全性应仅依赖**密钥的保密性**，而非算法的保密性；**算法公开可接受更充分的检验**，提升系统安全性
    4. **无绝对安全的密码系统**：任何密码系统的安全都可能因使用失误、技术突破、数学分析被破解
## 2. 对称密钥加密与非对称密钥加密
### a. 对称密钥加密
1. 定义与基础模型
	- **对称密钥加密** Symmetric Secret Key Cryptography 定义：
		- 也叫**共享密钥加密** Shared Key Cipher/**保密密钥加密** Secure Key Cipher
		- 加密与解密使用**相同或可通过简单变换相互推导**的密钥
		- **符合柯克霍夫原则**
	- **基础模型**：明文 → 共享密钥加密 → 密文 → 同一共享密钥解密 → 明文![Screenshot_20260316_141633](/blog/obsidian-assets/screenshot-20260316-141633-c9302d076d.jpg)
	- **缺陷**：通信双方必须**提前持有**共享密钥，存在**密钥分发难题**
2. **分组密码** Block Cipher 基础
	- **定义**：将输入比特流**划分为**固定长度的n**比特块**，**逐块独立加密**，块间无依赖关系
	- **常见参数**：块大小n常见64/128/256比特；密钥长度k常见40/56/128/192/256比特
	- **安全要求**：优质分组密码中，每个输出比特是所有n个输入比特和所有k个密钥比特的函数
	- **加密流程**：信息量一定会维持一致，不是压缩算法，即输入n bits一定输出n bits![Screenshot_20260316_141710](/blog/obsidian-assets/screenshot-20260316-141710-7b065faf1e.jpg)
3. **Feistel 费斯妥密码结构**
	- 1973年由IBM的Feistel提出，**几乎所有现代对称加密算法都基于该结构**
	- **设计目标**：通过**乘积密码逼近理想分组密码**，解决理想分组密码密钥过长的问题（理想key长度为$n\cdot 2^n$）
		- 要完整表示一个 n 位分组的可逆置换，需：
			1. 枚举所有 $2^n$ 个明文值，每个明文对应置换表的一行
			2. 为每个明文分配唯一的 n 位密文值，确保置换可逆
			3. 总比特数 = 明文数量 × 每个映射的比特数 = $n\cdot 2^n$
	- **安全设计**：通过两种机制实现**雪崩效应** avalanche effect
		- **扩散** Diffusion：让密文的统计特性**与明文的关系**尽可能复杂
		- **混淆** Confusion：让密文的统计特性**与加密密钥的关系**尽可能复杂
	- **设计要素**：
		- **块大小**：典型64/128比特，越大越安全
		- **密钥长度**：典型128比特，越大越安全
		- **轮数**：标准16轮，越多越安全
		- **子密钥生成算法**、**轮函数F**：复杂度越高抗密码分析能力越强
	- **加密和解密方法**：明文进入后，每轮将文字的**左侧**和**F函数+该轮对应密钥$K_i$加密**处理的**右侧异或**后变为**文字的右侧**，将**右侧原文直接变为左侧**，直到达到所要求的轮数![Screenshot_20260316_142059](/blog/obsidian-assets/screenshot-20260316-142059-e63f1c84c7.jpg)
4. 经典对称加密算法
	1. **DES** **数据加密标准** Data Encryption Standard
		- 1977年被美国国家标准局采用，基于Feistel结构，64比特明文块，56比特密钥
		- 特性：强雪崩效应，仅能通过暴力破解攻击
		- 缺陷：56比特密钥在算力指数提升后不再安全，1999年已可通过超算22小时破解
		- 总结构![Screenshot_20260316_142622](/blog/obsidian-assets/screenshot-20260316-142622-809adbb826.jpg)
		- **单轮结构**![Screenshot_20260316_142825](/blog/obsidian-assets/screenshot-20260316-142825-fb7de34dcf.jpg)
	2. **3DES** 三重DES
		- 采用E-D-E 加密-解密-加密流程：`C = EK3[DK2[EK1[P]]]`，密钥长度最高$56\times 3=168$比特
		- **向上兼容**：可通过设置K3=K2或K1=K2兼容DES算法，被PGP、S/MIME等互联网应用采用
		- 缺陷：软件实现速度慢，块大小仅64比特
	3. **IDEA算法**、**Blowfish算法**、**RC5算法**：均基于feistel结构
	5. **AES** **高级加密标准** Advanced Encryption Standard
		- 2001年美国发布的新一代加密标准，采用比利时学者Joan Daemen和Vincent Rijmen提出的Rijndael算法，替代DES与3DES，**不基于feistel结构**
		- 特性：块大小128比特，密钥长度128/192/256比特，免疫所有已知攻击，执行速度快、代码紧凑、设计简单
5. **分组密码工作模式**
	1. **定义**：描述如何重复应用分组密码的单块操作，安全处理长度超过一个块的数据
	2. **ECB** **电子密码本模式** Electronic Codebook
		- 逐块独立加密，**相同明文块会生成相同密文块**
		- **缺陷**：无法抵御重放攻击、密码本攻击，易通过密文规律泄露明文信息
		- **流程图**![Screenshot_20260316_143952](/blog/obsidian-assets/screenshot-20260316-143952-90519b262d.jpg)
	3. **CBC** 密文分组链接模式 Cipher-block chaining
		- 引入一次性初始化向量IV，**当前块的加密依赖前一块的密文**，每次对密文进行异或，相同明文块不会生成相同密文
		- **优势**：可抵御重放攻击和密码本构建攻击
			- **密码本构建攻击**：攻击者已知若干**明文 - 密文对（明密对）**，直接**遍历、匹配所有可能的完整置换密码本**，淘汰不符合明密映射的密码本，最终找到真实密码本（密钥）
		- **流程图**：![Screenshot_20260316_144050](/blog/obsidian-assets/screenshot-20260316-144050-fe3e2be5fe.jpg)
6. **流密码** Stream Cipher
	- 与分组密码不同，不将比特流划分为离散块，而是**将明文比特流**与**伪随机序列发生器**生成的**密钥流逐位异或**得到密文
	- 解密时使用**同一对称密钥生成相同伪随机序列**，再次异或即可还原明文
	- ![Screenshot_20260316_144415](/blog/obsidian-assets/screenshot-20260316-144415-d238022562.jpg)
7. **密钥分发问题与解决方案**
	- 对称加密的痛点：如何安全地在通信双方之间分发共享密钥
	- 常规分发方式：
		1. A选择密钥后**物理交付**给B
		2. **第三方选择**密钥后**物理交付**给A和B
		3. 双方使用**旧密钥加密传输新密钥**
		4. **可信第三方C通过加密链路**为A和B分发密钥
			- 典型解决方案：**密钥分发中心 KDC**
### b. 非对称密钥加密
1. 定义与基础模型
	1. **非对称密钥加密** Asymmetric/Public Key Cryptography **定义**：使用一对数学关联的密钥，**公钥** KU Public Key 可**公开分发**，**私钥** KR Private key 仅持有者**严格保密**；公钥加密的内容仅能通过对应的私钥解密，反之亦然
	2. **六要素**：明文、公钥、私钥、加密算法、密文、解密算法
	3. **安全基础**：**单向函数**，正向计算结果容易，反向逆运算在计算上不可行
	4. **安全要求**：
		1. 密钥对**生成、加密、解密**在计算上均**容易实现**
		2. 已知**公钥无法推导**出对应的**私钥**
		3. 已知**公钥和密文**，**无法还原**出原始**明文**
		4. **密钥对可互换**，同时支持加密与数字签名场景
	5. **发展历程**：
		1. 1976年，Diffie与Hellman在《密码学的新方向》中首次提出公钥密码思想
		2. 1977年，Rivest、Shamir、Adleman提出RSA算法
		3. 后续陆续出现ElGamal算法、椭圆曲线密码等方案
2. Diffie-Hellman DH **密钥交换算法**
	- **数学基础**：**离散对数难题**，大素数模下，**求幂取余计算容易，已知结果求指数的离散对数计算极难**
		1. **原根** Primitive Root：若p为素数，g是整数且满足：$g \mod p, g^2 \mod p, \dots, g^{p-1} \mod p$ 是**互不相同**的整数，且恰好**包含1到p-1**的所有整数，则称g是素数p的原根
		2. **原根g的唯一性**：对于1到p-1之间的**任意一个整数A**，都能找到**唯一的一个指数a**（a也在1~p-1范围内），使得等式 $A = g^a \mod p$ 成立
		3. **离散对数**：这个唯一的指数a，就被称为以g为底、模p时整数A的离散对数。类比普通对数，普通对数中$\log_g A = a$对应$g^a=A$；离散对数是在**模p**的有限域中，对应$g^a \mod p = A$
		4. **离散对数难题**：
			- **正向计算**，已知a求A：固定足够大的素数p和其原根g，如果知道指数a，计算$g^a \mod p$得到A，这个运算非常简单，算力消耗极低
			- **反向计算**，已知A求a：如果只知道结果A，要反推出对应的指数a，在p足够大的前提下，目前没有任何高效的算法能实现
		5. **引理**：$g^{ab} \mod p = (g^{a} \mod p)^b \mod p = (g^{b} \mod p)^a \mod p$
			- 证明：
				设$g^a=n\cdot p+i$
				则$g^a \mod p=i$，$g^{ab} \mod p = (n\cdot p+i)^b\mod p=i^b \mod p$
				故$g^{ab}\mod p=(g^a\mod p)^b\mod p$，同理另外可得证
	- **原理**：通信双方无需提前共享密钥，通过交换公开信息，协商出仅双方知晓的共享会话密钥
		1. 公开参数：大素数p、p的原根g
		2. Alice生成**私钥** private key $a$，计算公钥$A = g^a\mod p$，发送给Bob
		3. Bob生成**私钥**$b$，计算公钥$B = g^b \mod p$，发送给Alice
		4. Alice计算**共享密钥** key of exchanging $K = B^a \mod p = g^{ab} \mod p$
		5. Bob计算**共享密钥**$K = A^b \mod p = g^{ab} \mod p$
		![Screenshot_20260318_103512](/blog/obsidian-assets/screenshot-20260318-103512-9ac28a18f5.jpg)
	- **安全要求**：p至少为300位素数，a、b至少为100位，g无需过大，实践中常用2、3、5
	- **缺陷**：
		1. **仅支持密钥交换**，**无法**用于明文**加解密**（DH的过程只有两次加密计算，并没有反向解密，并且**私钥公钥难度不对称**）
		2. **无法抵御中间人攻击** Man-in-the-Middle Attack：攻击者可伪造双方公钥，分别与双方协商密钥，实现通信监听与篡改![Screenshot_20260318_103738](/blog/obsidian-assets/screenshot-20260318-103738-c30d723adc.jpg)
3. **RSA算法**
	1. 提出：1977年由MIT的Ron Rivest、Adi Shamir、Leonard Adleman提出，是目前应用最广泛的公钥密码算法
	2. **数学基础**：**大整数因数分解难题**，两个大素数相乘容易，将乘积分解回两个素数在计算上不可行，基于欧拉定理、费马小定理
		1. **欧拉函数**$φ(n)$ Euler's totient function
			$φ(n)$ 表示所有**小于**正整数n、且与n**互质**的正整数的个数
		2. 三大推论
			- **若n是素数**：$φ(n) = n-1$
				素数的性质是仅能被1和自身整除，因此所有小于n的正整数都与n互质，个数为n-1
			- **若n是合数**：鸽笼定理
				先将n分解为不同素数的幂次乘积：$n = p_1^{a_1} \cdot p_2^{a_2} \cdot \dots \cdot p_k^{a_k}$，则$φ(n) = n \cdot (1-\frac{1}{p_1}) \cdot (1-\frac{1}{p_2}) \cdot \dots \cdot (1-\frac{1}{p_k})$
			- **互质两数的乘积推论**：
				若p和q互质，则 $φ(pq) = φ(p) \cdot φ(q)$
				**特殊情况**：若p、q是两个不同的素数，则$φ(pq)=(p-1)(q-1)$
		3. **欧拉定理** Euler's theorem / 费马-欧拉定理
			1. 定理条件与结论
				- **条件**：n和a是互质的正整数 $gcd(a,n)=1$
				- **结论**：$a^{φ(n)} \equiv 1 \pmod{n}$
			2. **证明**：构造互质集合的置换，通过乘积相等推导出结论
				1. 定义集合
					设S是所有小于n且与n互质的正整数的集合，即 $S=\{a_1,a_2,\dots,a_{φ(n)}\}$，集合元素个数为$φ(n)$，取任意一个数$c \in S$
				2. 证明对S中任意元素$a_i$，$c \cdot a_i \equiv a_j \pmod{n}$，有$a_j \in S$
					**原因**：$c$和$a_i$都与n互质，它们的乘积也与n互质，因此结果必然是S中的某个元素
				3. **证明置换性质**：若$c \cdot a_i \equiv c \cdot a_j \pmod{n}$，则$a_i = a_j$
					**原因**：c与n互质，模n运算中可消去c，因此不会出现不同$a_i$对应相同乘积的情况，即$\{c \cdot a_1, c \cdot a_2, \dots, c \cdot a_{φ(n)}\}$**是S的模n置换**
				4. 两边取集合所有元素的乘积，由置换性质右式满足，得到$\prod_{k=1}^{φ(n)} (c \cdot a_k) \equiv \prod_{k=1}^{φ(n)} a_k \pmod{n}$
				5. 左边乘积可提取$φ(n)$个c的公因子，即：$c^{φ(n)} \cdot \prod_{k=1}^{φ(n)} a_k \equiv \prod_{k=1}^{φ(n)} a_k \pmod{n}$
				6. 集合S中所有元素都与n互质，因此它们的乘积也与n互质，模n运算中可消去该乘积项，最终得到：$c^{φ(n)} \equiv 1 \pmod{n}$
		4. **费马小定理**：是欧拉定理的特殊情况，当n为素数时，$φ(n)=n-1$，代入欧拉定理可得$a^{n-1} \equiv 1 \pmod{n}$（a与n互质）
	3. **流程**：使用时根据n将原文**分块**，每块m都满足$m<n$
		1. **密钥生成**
			- 选择两个大素数p、q，计算$n = p\cdot q$
			- 计算欧拉函数$φ(n) = (p-1)(q-1)$
			- 选择整数e，满足$1<e<φ(n)$，且$e$与$φ(n)$互质
			- 用**中国剩余定理**计算d，满足$e\cdot d ≡ 1 \mod φ(n)$，d为e的模$φ(n)$**乘法逆元**
			- 公钥为$(e, n)$，私钥为d
		2. **加密**：明文$m < n$，密文$c = m^e \mod n$
		3. **解密**：明文$m = c^d \mod n$
		4. **正确性证明**：
			1. 若m,n**互质**，有$m^{ed}\mod n=m^{kφ(n)+1}\mod n=m$
			2. 若m,n**不互质**，m**一定为p或q的倍数**，若两者皆为，不满足$m<n$
				- **若m不是p的倍数**： $\gcd(m, p) = 1$ ，由费马小定理  $m^{p-1} \equiv 1 \pmod{p}$ ，则$m^{ed} = m^{k(p-1)+1} = (m^{p-1})^k \cdot m \equiv 1^k \cdot m \equiv m \pmod{p}$
				- **若m是p的倍数**： $m \equiv 0 \pmod{p}$ ，则  $m^{ed} \equiv 0^{ed} \equiv 0 \equiv m \pmod{p}$
				综上均有  $m^{ed} \equiv m \pmod{p}$ ，同理可证$m^{ed} \equiv m \pmod{q}$
				因为  p, q  为互质质数，由  $m^{ed} \equiv m \pmod{p}$  且  $m^{ed} \equiv m \pmod{q}$ ，可得$m^{ed} \equiv m \pmod{pq} \implies m^{ed} \equiv m \pmod{n}$
		![Screenshot_20260318_110538](/blog/obsidian-assets/screenshot-20260318-110538-d976dc59f6.jpg)
	4. **安全性分析**：
		1. 最有效的攻击方式是**对n进行因数分解**，分解出p、q即可计算出私钥d
		2. RSA安全性建立在正向计算得到n容易，反向计算分解n为p和q较难，**攻击时间本质上等于大整数分解的时间**。只要n的长度足够大，分解n的成本会随位数指数级增长
		3. 1024位RSA已被NIST在2011年弃用，2014年起禁用；目前**主流使用2048位RSA**，可预见的未来内难以被破解
		4. 存在主动**伪造公钥**的欺骗攻击，需通过数字签名验证公钥真实性
4. **公钥密码**的三大**应用场景**
	1. **加解密**：发送方用接收方的公钥加密消息，仅接收方可用对应私钥解密，解决密钥分发难题
	2. **数字签名**：发送方用自己的私钥对消息签名，接收方用发送方的公钥验证签名，解决身份认证、防篡改与防抵赖问题
	3. **密钥交换**：通信双方协作协商会话密钥，无需提前共享秘密
5. **对称与非对称加密的差异**
	1. **密钥特性**
		- 对称加密：加密解密使用**同一密钥**
		- 非对称加密：加密解密使用**公钥+私钥对**，二者数学关联但无法互推
	2. **优势**
		- 对称加密：加密**速度快**、计算成本低，有成熟的低成本VLSI芯片
		- 非对称加密：无需提前共享密钥，彻底**解决密钥分发难题**，支持数字签名与身份认证
	3. **缺陷**
		- 对称加密：密钥分发困难，多用户通信场景下密钥管理复杂度极高
		- 非对称加密：计算开销大、加密速度慢，硬件成本高，不适合大量数据加密
	4. **典型算法**
		- 对称加密：DES、3DES、AES、IDEA、Blowfish、RC5
		- 非对称加密：RSA、Diffie-Hellman、ElGamal、椭圆曲线密码
6. 实际工程应用与常见误解澄清
	1. 主流应用方案：**混合加密体系**
		- 用**非对称加密算法**（如RSA）安全**分发对称加密的会话密钥**
		- 用**对称加密算法**（如AES）对**大量业务数据**进行加解密，兼顾安全性与加解密效率
	2. **常见误解澄清**
		1. **公钥加密在防范密码攻击上比对称加密更安全**
			- 错误，两者的安全性**均依赖于密钥长度与解密的计算工作量**，从抗密码分析角度**无绝对优劣**
		2. **公钥加密会让对称加密过时**
			- 错误，公钥加密的**计算开销极大**，仅适用于密钥管理与数字签名场景，无法替代对称加密在大量数据加解密中的作用

# 三、数字签名
## 1. 数字签名基础
1. **数字签名** Digital Signature **背景**
	1. **概念**：弥补加密/解密无法防范的**欺骗问题**，保障通信中的不可否认性与身份认证
	2. **密码学** Cryptography 的**作用与局限性**
		1. 密码学**解决**的问题：保障通信**内容的机密性**，例如A与B之间的通信内容无法被第三方获取
		2. 密码学**无法解决**的问题：**无法防范**通信双方的**欺骗纠纷**，仅靠加解密无法解决消息真伪与发送/接收的**抵赖问题**
		3. **通信纠纷场景**：Alice向Bob发送消息后，可能出现两类争议：
			1. Bob**伪造**不同消息，声称是从Alice处接收的原始消息
			2. Alice**否认发送**过该消息，且Bob无法证明Alice的发送行为
	3. 现实生活中的纠纷解决思路
		1. 法律层面的合同效力：如何让法庭认可各类契约/消息的真实性，解决纠纷举证问题
		2. 组织审批的证明问题：完成上级/组织审批流程后，如何有效证明已获得合法审批权限
	4. 数字签名的**现实意义**：作为解决上述纠纷的技术手段，替代现实中的手写签名、公章审批等方式，实现电子场景下的不可否认、身份认证与不可篡改
	5. **手写签名的特征**：与印章 seal，指纹 fingerprint 类似，数字签名需对标
		- **真实性** credible：签名可信，接收方确信签名者已正式签署文档
		- **不可伪造** fabricated：无法伪造合法签名
		- **不可复用** reusable：签名与签署内容绑定，无法重复使用
		- **不可篡改** be changed：签署后的文档无法被修改
		- **不可否认** undeniable：签名者无法抵赖签署行为
		- **时间绑定** bound：部分场景下签名与签署时间强关联
		- **法律有效性** legal evidence：签名可作为合法证据，且可被验证
	6. 常用算法
		- DSS/DSA：美国国家标准，基于离散对数难题，政府主推
		- RSA：行业事实标准，支持度广
		- 椭圆曲线算法
2. 数字签名的**需求与特征**
	1. **需求**：**手写签名的数字化替代**
		- 手写签名无法直接应用于电子文档场景，因此需构建数字签名体系解决电子签署的信任问题
	2. **数字签名系统的要求**
		1. **内容绑定**：签名需与签署文档强关联，确保文档内容无法被篡改
		2. **可验证性**：与不可伪造性接收方可验证签名真伪，第三方无法伪造有效签名
		3. **不可否认性**：签名者无法抵赖自身的签署行为
		4. **第三方可验证性**：第三方机构可对签名进行核验，用于解决签署纠纷
		5. **多维度验证**：可验证签名的签署主体、时间、内容等关键信息
	3. **数字签名的要求**
		1. **与消息强关联**：签名依赖于签署消息的内容，防止消息被篡改
		2. **唯一性与不可抵赖**：签名需使用发送方独有的信息生成，避免伪造与抵赖
		3. **高效性**：签名的生成、识别、验证流程需操作简便，且可在短时间内完成
		4. **不可伪造性**：伪造难以计算
		5. **易保存性**：生成的签名不应太大
3. **数字签名算法**：**RSA签名**
	1. 前置准备：**RSA密钥对生成**
		1. 选取两个大质数 $p$、$q$
		2. 计算模数：$n = p \times q$
		3. 计算欧拉函数：$\phi(n) = (p-1) \times (q-1)$
		4. 选取公钥指数 $e$：满足 $1 < e < \phi(n)$ 且 $\gcd(e, \phi(n))=1$
		5. 计算私钥指数 $d$：满足 $d \times e \equiv 1 \pmod{\phi(n)}$
		6. 密钥对定义
			- 公钥：$(e, n)$ 公开用于验签
			- 私钥：$(d, n)$ 保密用于签名
	2. 签名**生成**流程：**发送方操作**
		1. **消息哈希预处理**
			- 选择密码学单向哈希函数 One-way hash algorithm 如SHA-256，对原始消息 m 计算哈希值 h
			- **为什么要使用哈希函数**
				1. 基础示例中，要求待签名文档 m 必须为 $[0, n-1]$ 区间内的整数，无法直接适配任意格式的电子文档
				2. 直接对超长文档进行RSA模幂运算，计算开销极大，效率低下
			- **单向哈希算法** One-way Hash Algorithm **特性**
				1. **单向性**：由文档**计算哈希值容易**，由哈希值**反推**原始文档**不可行**
				2. **固定长度**：无论输入长短，输出哈希值长度固定
				3. **抗碰撞性**：难以找到两个不同文档，使其哈希值相同
				4. 实际工程中常用算法：SHA-256、SHA-512等，淘汰MD5、SHA-1等存在碰撞漏洞的算法
					- **碰撞漏洞**是指密码学哈希函数中存在两个**不同的输入数据**，经过哈希计算后，得到了**完全相同的哈希值**的现象
			- 常见哈希算法：
				- 经典：MD4、MD5、SHA-0/SHA-1/SHA-2（SHS安全哈希标准）、HAVAL、RIPEMD
				- 新一代：SHA-3 由Keccak算法胜出NIST竞赛，2015年成为FIPS标准
		2. **私钥模幂签名**
			- 公式：$S = h^d \pmod n$
			- 其中 S 为生成的数字签名，与原始消息$m$一同发送给接收方
	3. 签名**验证**流程：**接收方操作**
		1. **重新计算消息哈希**
			- 对收到的原始消息 m 使用**相同哈希函数**计算哈希值 $h'$
		2. **公钥模幂验签**
			- 公式：$h_s = S^e \pmod n$
			- 数学原理：$(h^d \pmod n)^e \pmod n = h^{d \times e} \pmod n = h^1 \pmod n = h$
		3. **哈希值比对**
			- 若 $h_s = h'$，则验证通过，确认：
				1. **完整性**：消息自签名后未被篡改
				2. **真实性**：签名由持有对应私钥的主体生成
				3. **不可否认性**：签名者无法抵赖
			- 若 $h_s \neq h'$，则验证失败，消息可能被篡改或签名无效

4. **MD5算法**
	1. 算法整体架构
		MD5算法流程分为两大阶段：**填充** Padding 与**哈希运算** Hashing，最终输出128位固定长度的散列值
		1. **填充阶段**
			- 目的：将输入数据长度补为**512位的整数倍**，为后续分组运算做准备
			- 具体规则：
				1. 追加比特1
				2. 追加若干比特0，直至长度距离512位的整数倍还差64位
				3. 追加原始输入数据的长度，用64位表示
		2. **哈希运算阶段**
			- 输入规模：处理512位的**分组数据**
			- 运算逻辑：
				1. 初始化128位的链接变量 A、B、C、D，拆分为4个32位整数
				2. 对每个512位分组进行4轮共64步的非线性迭代运算
				3. 每轮使用不同的非线性逻辑函数、循环常数及消息子分组，最终将运算结果与初始链接变量相加得到最终摘要
		3. 运算逻辑
			1. **非线性逻辑函数**
				MD5定义了4轮不同的非线性函数 F, G, H, I，每轮16步，核心依赖按位运算：
				- F 非线性逻辑
				- G 非线性逻辑
				- H 异或，线性
				- I 混合逻辑
				注：$\land=与, \lor=或, \oplus=异或, \neg=非$
			2. **基础运算单元**
				- **模 $2^{32}$ 加法**：所有32位字长的运算结果均对 $2^{32}$ 取模，确保数据溢出不丢失
				- **循环左移** `<<<`：将32位数据循环左移指定位数，保证扩散性
				- **常数与消息调度**：每轮使用固定的循环常数 $K_i$ 和经过打乱的消息子分组 $M_j$，增加算法复杂度
		4. **流程图**![Screenshot_20260323_145622](/blog/obsidian-assets/screenshot-20260323-145622-3ee27fd17a.jpg)
	2. 安全缺陷：**抗碰撞性失效**
		- 现状：已被证实存在高效的碰撞攻击算法，如王小云院士提出的攻击方法，可在多项式时间内找到两个不同的输入，使其哈希值完全相同
		- 后果：
			1. **无法保障数据完整性**：攻击者可篡改数据但保持MD5哈希值不变，绕过校验
			2. **数字签名伪造**：结合RSA等签名体系，可伪造合法签名，导致不可否认性失效
5. **单向哈希算法的攻击**
	1. **原像攻击与原像抵抗性** Preimage Resistance
		- 攻击目标：给定哈希值 h，找到任意一个输入消息 x，满足$\text{Hash}(x) = h$
		- 抵抗性要求：算法需具备原像抵抗性，即由哈希值反推原始输入在计算上不可行
		- 安全意义：保障哈希值的单向性，防止攻击者根据散列值还原敏感原始数据
	2. **第二原像攻击与第二原像抵抗性** Second Preimage Resistance
		- **攻击目标**：给定原始消息 $x_1$，找到另一个不同消息 $x_2$（$x_1 \neq x_2$），满足 $\text{Hash}(x_1) = \text{Hash}(x_2)$
		- 抵抗性要求：算法需具备第二原像抵抗性，即无法找到与指定消息哈希值一致的替代消息
		- 安全意义：保障数据完整性，防止攻击者篡改原始消息内容但保持哈希值不变
	3. **碰撞攻击与抗碰撞性** ****
		- **攻击目标**：直接找到任意两个不同的消息 $m_1$ 和 $m_2$（$m_1 \neq m_2$），满足$\text{Hash}(m_1) = \text{Hash}(m_2)$
		- **抵抗性要求**：算法需具备抗碰撞性，即难以找到任意一对哈希值相同的不同输入
		- **扩展攻击**：**前缀碰撞攻击**
			- 目标：给定两个不同前缀 $p_1$、$p_2$，找到后缀 $s_1、s_2$，使 $\text{Hash}(p_1 \parallel s_1) = \text{Hash}(p_2 \parallel s_2)$ $\parallel$ 为拼接操作
			- 危害：可构造合法+恶意的差异化内容，保持哈希值一致，绕过完整性校验
	4. **安全强度逻辑关系**
		- **难度层级**：碰撞攻击 > 第二原像攻击 > 原像攻击 抗碰撞性要求最高，原像抵抗性要求最低
		- **推导逻辑**：若算法**满足抗碰撞性**，则**必然满足第二原像抵抗性和原像抵抗性**；但反之不成立。因此，抗碰撞性是单向哈希算法最核心的安全指标
	5. 实际应用危害：因MD5不满足抗碰撞性和第二原像抵抗性，攻击者可轻松实施前缀碰撞攻击：
		1. 构造两个不同后缀，分别拼接至合法前缀和恶意前缀，生成哈希值相同的文件
		2. 接收方校验哈希值时判定数据合法，实际接收的是篡改后的恶意文件，导致数据完整性和签名有效性彻底失效
6. **消息认证码** MAC Message authentication code
	1. 基本定义：又称**带密钥的哈希函数**，提供消息完整性验证服务，仅收发双方共享密钥
	2. 工作流程
		- **发送方**：用共享密钥+MAC算法对消息生成MAC值，随消息发送
		- **接收方**：用相同密钥+算法重新计算MAC，与接收值比对，一致则消息完整且真实
		- 流程图![Screenshot_20260325_101650](/blog/obsidian-assets/screenshot-20260325-101650-70161d37b1.jpg)
	3. 常见算法
		- HMAC：基于哈希函数的MAC，如HMAC-MD5、HMAC-SHA1、HMAC-SHA256
		- CBC-MAC：基于分组密码CBC模式的MAC
		- 其他：UMAC、CMAC、VMAC、Poly1305-AES等
	4. **HMAC 算法**
		- $HMAC(K,m) = H((K⊕opad) || H((K⊕ipad) || m))$，opad为外填充常量、ipad为内填充常量，$⊕$为异或，$||$为连接
		- 抗碰撞性优于单纯哈希函数，HMAC-MD5/SHA1 **目前为止仍安全**，是IPSec、TLS核心组件
	5. **CBC-MAC** **密码分组链接消息认证码**
		1. 底层基础：基于分组密码的加密算法$E()$实现，需收发双方共享密钥k
		2. 计算前提：将待认证明文m切分为等长分组$(m_1,m_2,...,m_x)$，分组长度与$E()$输入一致；初始化向量固定为0（$0x00000000….00$）
		3. 运算方式：采用链式加密方式，先以初始0值与首个分组做运算，后续每轮以上一轮加密结果与当前分组运算后再加密，最终加密结果即为MAC值
		4. 流程图：![Screenshot_20260325_102534](/blog/obsidian-assets/screenshot-20260325-102534-42e83fc8a6.jpg)
	5. **与数字签名的区别**
		- **密钥**：MAC需收发双方**共享密钥**，数字签名无需共享，用公钥私钥对
		- **验证范围**：MAC仅**特定接收方可验证**，数字签名所有持有发送方公钥者均可验证
		- **无不可否认性**：MAC无此特性，验证者也能生成MAC，而数字签名具备
		- **效率**：MAC运算远快于数字签名，适用于无需不可否认性的场景
		- 应用选择
			- 不用全程用数字签名：MAC运算远快于数字签名，且实际场景中并非都需要不可否认性
			- 需高效验证、无纠纷举证需求时用MAC（防止宇宙射线干扰传输）；需不可否认性、多方可验证时用数字签名
	6. 与加密的组合方式
		- **仅认证**：直接对明文生成MAC并验证
		- **认证+保密**（绑定明文）：先对**明文生成MAC**，再将**明文+MAC**一起**加密**
		- **认证+保密**（绑定密文）：先**加密明文**得到密文，再对**密文**生成MAC并**验证**
		- **流程图**：![Screenshot_20260325_102903](/blog/obsidian-assets/screenshot-20260325-102903-90db91016c.jpg)
7. **PGP** Pretty Good Privacy
	1. **基本特性**
		- 广泛用于**邮件和文件存储安全**，提供**数字签名、完整性验证、信息加密、数据压缩**等服务
		- **兼容邮件格式**、支持数据分块，跨平台
		- 早期免费，现商用需付费，源代码免费，基于成熟加密算法(**对称、非对称**都支持)
		- **算法组合**：RSA/Diffie-Hellman 用于密钥交换，IDEA/3DES/AES 用于对称加密，MD5/SHA-1 用于哈希校验
	2. **安全模式**
		- **仅认证**：发送方用**私钥**对消息哈希值**签名**，接收方用公钥验证
		- **仅保密**：生成**会话密钥加密消息**，用接收方的**公钥加密会话密钥**，生成数字信封，将加密后的消息与数字信封一起发送给接收方
		- **保密+认证**：先用**发送方私钥签名哈希值**，再用**会话密钥加密消息+签名**，最后用接收方的**公钥加密会话密钥**
		- 流程图：$KR_a$ 私钥 $K_s$ 会话密钥 $KU_b$ 接收方公钥 ![Screenshot_20260325_103846](/blog/obsidian-assets/screenshot-20260325-103846-c3e898019f.jpg)
## 2. 身份认证与访问控制
1. **身份认证** Authentication
	1. **概念**
		- **问题**：验证**你是你所声明的身份**，是所有访问控制的前提
		- **认证依据** / **认证因子**：
			- **拥有的物品**：身份证、智能卡、手机
			- **知道的信息**：密码、密保问题
			- **所处位置**：IP地址
			- **自身特征**：
				- **生物特征** Biometrics characteristics：指纹、掌纹、虹膜、面部轮廓、DNA等
				- **行为特征** Behavioral characteristics：笔迹、声纹、按键节奏、走路姿态等
		- **研究方向**：
			- **认证技术** Authentication Technology：密码认证，生物特征认证
			- **认证协议** Authentication Protocol：挑战-响应认证，KERBEROS认证
	2. **基于密码的身份认证**
		1. **基本流程**
			1. **系统存储已认证**用户的账号与密码文件
			2. 用户持有**私密密码**，系统通过**校验密码**完成身份认证
		2. **安全隐患**
			1. 密码在用户与系统间传输时，**易被窃听**
			2. 密码**文件本身**难以做到**保密**存储
			3. **易记**的密码往往易被猜测，存在**猜解风险**
			4. 系统的**密码校验环节**存在安全漏洞
		3. **安全影响因素**
			1. 攻击者**密码尝试的频率/次数限制**
			2. 密码的**存储形式**：明文/哈希值
			3. 密码**在网络中的传输形式**：明文/哈希值等
			4. 密码**修改流程**：修改时如何验证用户身份
			5. 密码**生命周期**：密码的修改频率
			6. 其他细节规则
				- 屏幕不显示明文密码，以 `*` 替代
				- 定义密码的长度与格式规范
				- 关键操作时要求重新输入密码
				- 短时间内多次输错密码，禁止账号登录
		4. **密码存储的演进**
			1. **原始阶段**：明文存储，易泄露，如2011年CSDN密码泄露
			2. **早期阶段**：存储密码哈希值$H(passwd)$，验证时比对$H(p’)$与$H(p)$，基于单向哈希函数
				1. 早期 Unix 系统采用 **DES** Data Encryption Standard 算法作为密码的单向哈希函数，具体流程如下：
					1. **明文截断**：将用户输入的密码**截取前 8 个字符**ASCII码，作为 DES 加密的密钥
					2. **明文加密**：使用该密钥对一个固定的明文，通常是**全 0 字节**，即  `NUL` 进行 DES 加密，为了增加密码的安全性，Unix 系统引入了一种人工优化的迭代机制，将上述 DES 加密过程重复执行 25 次
					3. **结果处理**：加密后的 64 位二进制数据会被编码成字符串，最终存储在 `/etc/passwd` 文件中
				2. **安全隐患**
					1. 从理论上计算，若密码字符集包含 52 个大小写英文字母，10 个数字，32 个常见符号，总计 94 个可打印字符，一个 8 位密码的组合总数为$94^8 \approx 6.09 \times 10^{15}$
					2. 但人的**密码使用习惯**严重削弱了安全性
						- **密码偏好**：用户倾向于使用字典单词、人名、宠物名、生日等易于记忆的内容
						- **实际空间**：这类密码的组合量仅在 100万量级
						- **后果**：这使得攻击者可以通过**字典攻击** Dictionary Attack  或**彩虹表** Rainbow Table 高效破解大量密码
					3. **存储设计缺陷**：公开可读
						- Unix 系统最初将所有用户信息，包括加密后的密码哈希值，存储在 `/etc/passwd` 文件中，该文件**默认对所有用户都是可读的**
						- **攻击路径**：任何本地用户都可以读取该文件，获取完整的密码哈希列表，然后在**本地进行离线暴力破解**，无需与服务器交互
			3. **加盐** Salting：升级1，密码设置时**生成随机盐值**，存储$H(盐值+密码)$，相同密码对应不同哈希值，提升字典攻击难度，12位盐值对应$2^{12}$种哈希结果
				1. 早期 Unix 密码系统中，两个用户若使用相同的密码，密文记录完全一致
					- **彩虹表/预计算攻击**：攻击者只需**预先计算**一次所有常用字典单词的**哈希值**，即可一次性破解系统中所有用户的密码
					- **通用性**：所有 Unix 机器使用相同的哈希函数，一旦密码被破解，威胁可全面扩散
				2. 例子：以 `shmat: fURxfg, 4hLBX: 14510: 30: Vitaly: /u/shmat: /bin/csh` 为例
					- 密码字段： `fURxfg, 4hLBX`
					- 前 2 位 `fU`：盐 Salt。由系统随机生成的 2 个字符，共 12 比特信息，涵盖 $2^{12}$ 种组合
					- 后 11 位 `Rxfg, 4hLBX`：实际哈希值
				3. **流程**
					1. **生成盐值**：当用户首次设置密码时，系统随机生成 12 bit 的盐值
					2. **混合哈希**：将盐值与用户密码拼接或混入，再共同进行单向哈希计算
					3. **存储**：将盐值与哈希结果拼接后，一起存入密码文件
				4. **攻击成本**：
					- **无盐攻击成本**：$O(N)$ 攻击者只需构建一张包含所有字典单词的哈希表，遍历一次即可匹配所有用户
					- **加盐攻击成本**：$O(N \times S)$ 攻击者必须针对每一个盐值，重新计算一次所有字典单词的哈希
			4. **影子密码**：升级2，哈希密码存储在**仅管理员可读**的 `/etc/shadow` ，原 `/etc/passwd` 仅存用户基本信息，新增**密码过期时间**
				1. 新格式 `/etc/passwd` 示例
					以 `shmat: x: 14510: 30: Vitaly: /u/shmat: /bin/csh` 为例：
					- 密码字段：由原密文变为 `x`
					- 含义：这是一个占位符，指示系统密码实际存储在 **影子文件** 中
				2. 存储位置 `/etc/shadow`
					系统管理员 root 将加密后的密码哈希、盐值等敏感信息写入`/etc/shadow`
					- **权限控制**：该文件仅 root 用户可读
					- **安全增益**：普通用户无法接触到密码哈希，彻底阻断了本地用户获取密文进行离线破解的路径
				3. **密码过期策略**
					- **过期日期设置**：为密码设定具体的有效期
					- **强制改密**：用户登录后，若密码过期，系统将强制要求其修改密码，否则限制后续操作
		5. **密码的常见问题与防护**
			- **十大错误**：
				1. 空密码或使用默认出厂密码
				2. 使用简单易猜的单词作为密码
				3. 使用喜爱的电影明星、偶像等公众人物姓名
				4. 使用伴侣、家人、宠物等亲近关系的人名
				5. 所有平台、账号共用同一个密码
				6. 将密码写在便利贴上并贴在办公设备旁
				7. 将密码清单贴在键盘下方或抽屉里
				8. 将密码存储在电子表格、PDA 或纸质通讯录/名片夹中
				9. 将密码记录在个人日记、笔记本等纸质载体
				10. 向自称系统管理员的陌生人主动提供密码
			- **密码猜测手段**：
				1. **自动化暴力破解**
					- **默认密码**：尝试使用各类软件、设备出厂时预设的默认账号密码
					- **短密码枚举**：暴力遍历 1~3 个字符的所有组合，数字、小写字母等
					- **字典攻击**：调用**电子词典**，尝试所有常见单词、常见组合，如  password ,  123456
				2. **基于个人信息的社会工程学** Social Engineering
					- **泄露信息**：尝试用户的手机号、身份证号、门牌号、生日等个人公开信息
					- **熟人攻击**：尝试用户的亲友、宠物、偶像名字等被泄露的个人关系数据
				3. **技术拦截与监听**
					- **键盘监听**：通过木马程序 Trojan 在本地窃取键盘敲击记录 Keylogger
					- **通讯截获**：劫持网络通讯流量，抓取明文或弱加密的密码数据包
				4. **社交工程**
					- **伪装身份**：冒充客服、IT 管理员、银行人员，通过电话、邮件诱导用户主动说出密码
					- **防护方法**：复杂密码, 字符+数字、不同场景用不同密码、防范木马/DNS攻击、不使用不可靠设备/第三方密码软件、不在网吧登录重要服务
	3. 生物/行为特征认证
		1. **常见类型**
			- **生物特征**：指纹、虹膜、视网膜、人脸、手掌几何特征、静脉、DNA
			- **行为特征**：笔迹、步态、打字节奏、鼠标手势、声纹
			- **新兴特征**：脑电波、心电图、指静脉，及分析生理特征的意图生物识别（预测危险行为）
		2. **指标**
			- **误判率** FAR False Accept Rate：接受恶意用户，即冒名通过
			- **拒登率** FRR False Reject Rate：拒绝合法用户，即合法用户无法通过
			- **等错误率** EER Equal error rate：FAR=FRR时的数值，**越小算法精度越高**，虹膜扫描EER<0.0001%，为目前最优
		3. **优缺点**
			- **优点**：无法被盗、丢失、遗忘
			- **缺点**：设备/维护成本高、算法存在精度误差、存在仿冒风险、隐私问题
			- **现状**：密码虽有缺陷，但因易用性，短期内难以被完全替代
		4. **关键评估维度**
			- **普适性** Universality：生物特征在人群中的普遍存在程度
			- **唯一性** Uniqueness：生物特征区分不同个体的能力
			- **持久性** Permanence：生物特征随时间和年龄变化的稳定性
			- **采集性** Collectability：生物特征数据采集的难易程度
			- **性能表现** Performance：生物特征识别系统的准确率、速度与鲁棒性
			- **可接受度** Acceptability：公众对使用该生物特征的接受程度
			- **抗欺骗性** Circumvention：生物特征系统抵御伪造与欺骗攻击的难度
	4. 网络身份认证
		1. **基础协议的问题** v1.0-v4.0
			- 简单身份声明：直接复制
			- 绑定IP：虚拟伪造IP
			- 传输密码/加密密码：**均易遭遇重放攻击**，攻击者记录并重发认证信息
			- **重放攻击**：攻击者截获历史合法认证报文，**原样复用、延后重发**，不参与实时通信链路
		2. **挑战-响应认证**
			- 目标：抵御重放攻击，使用**一次性随机挑战码**
			- v5.0 **对称密钥版**：Bob发送随机数R，Alice用共享密钥加密R并返回，验证通过则证明Alice为合法用户
			- v6.0 **公钥版**：Alice用私钥加密R，Bob用Alice公钥解密得到R，解决**共享密钥泄露问题**

		3. **KERBEROS认证**
			1. 背景
				- 解决分布式环境下的客户端-服务器认证，抵御**冒名访问** Spoofing、**IP伪造** Address Spoofing、**重放攻击** Replay Attack 三大威胁
				- **设计目标**：
					- 安全性 Safety：攻击者无法获取足够信息来假冒其他用户
					- 可靠性 Reliability：无法获得 Kerberos 服务即无法获取所需服务，需高可靠性
					- 透明性 Transparency：用户无需感知认证服务的存在
					- 可扩展性 Scalability：系统能支持大量用户与服务器
				- **实现基础**：基于**对称加密**，分v4和v5两个版本
				- 基于常规加密方式，**不使用公钥加密**
			2. KERBEROS **v4** 协议演进
				1. 概念与标识
					1. 实体概念
						- C Client：**访问服务的客户端主机**/**用户端**
						- AS Authentication Server：**认证服务器**，存储用户密码，负责初始身份认证
						- TGS Ticket Granting Server：**票据授予服务器**，负责发放访问会话票据
						- V Application Server：**提供具体应用服务的服务器**
					2. 标识与信息
						- IDc：用户在客户端C上的唯一标识符
						- IDv：应用服务器V的唯一标识符
						- Pc：客户端C上用户输入的密码
						- ADc：客户端C的网络地址
						- Kv：认证服务器AS与应用服务器V之间共享的密钥
				2. **最简版**：
					1. **流程**：客户端C向AS提交身份+密码+服务器ID，AS返回票据，C持票据访问应用服务器V
					2. **问题**：
						1. **频繁密码交互**：要求用户持续输入密码进行身份验证，体验差且存在泄露风险
						2. **Ticket 复用率低**：同一服务器在不同时间请求服务，或访问不同服务器时，均需申请新的票据，效率低下
						3. **明文传输密码**：用户密码在网络传输过程中以明文形式出现，极易被监听窃取
						4. **Ticket 易被重放**：票据一旦被攻击者截获，即可伪造客户端身份发起重放攻击，安全性不足
					3. **流程图**：![Screenshot_20260330_145247](/blog/obsidian-assets/screenshot-20260330-145247-a992e4f9f4.jpg)
				2. **改进版**：
					1. **流程**：增加TGS，登录时从AS获取TGS票据TGT（**一次登录**），访问服务时从TGS获取服务票据ST（**一次服务**）
						1. 客户端发起登录请求，向AS发送**自身ID**、**TGS ID**、**时间戳**，申请TGT
						2. AS 验证密码&身份后，返回**会话密钥 Kc-tgs** 和 **TGT** Ticket Granting Ticket，TGT封装了：Kc-tgs、客户端ID、TGS的ID、有效期等，**用TGS的密钥加密**，客户端无法解密篡改
						3. 客户端需访问服务时，向TGS发送**目标服务ID**、**TGT**、**用Kc-tgs加密的认证符**，含客户端ID+时间戳
						4. TGS验证TGT和认证符合法后，返回**服务会话密钥Kc-v**和**ST服务票据**，ST封装了：Kc-v、客户端ID、服务ID、有效期等，用**对应服务的密钥加密**
						5. 客户端持**ST**和**用Kc-v加密的新认证符**访问目标服务，服务验证通过后授权访问
					2. **改进**：无明文传输、减少密码输入、票据增加有效期防重放
						- **一次登录**：仅登录时向AS验证密码获取TGT，后续无需重复输密码，实现SSO
						- **一次服务**：每访问一个新服务，需用TGT向TGS申请对应ST，单张ST仅对应单个服务使用
					3. **流程图**：![Screenshot_20260330_145818](/blog/obsidian-assets/screenshot-20260330-145818-87e10cec03.jpg)
					4. **问题**：
						1. **有效期** Lifetime 的两难
							- 票据有效期若太短，用户需频繁重新输入密码，体验差
							- 若太长，重放攻击 Replay Attack 的风险窗口巨大
						2. **身份认证的缺失**
							- **缺双向认证**：无法证明持票人就是申请人，也无法证明服务器V是真的，容易被冒充
				3. **最终版**：增加**认证器** Authenticator，包含**时间戳**、**客户端地址**，服务器返回**时间戳+1**验证，完成**双向**认证
			3. Kerberos 安全性与优点
				1. **正确性**
					- Kerberos 的正确性已被形式化证明
					- 协议模型依赖**安全的时间服务**
				2. **可信第三方**
					- 引入可信第三方 KDC，统一负责认证与票据发放
					- 共享密钥的分发与管理变得更简单
					- Kerberos 服务器**减轻了应用服务器的安全负担**
					- 与安全相关的数据集中控制与保护，使攻击更困难
				3. **票据** Ticket
					- 可将 AS 的认证结果与会话密钥安全传递给应用服务器
					- 票据在有效期内**可重复使用**，降低开销，使用更方便
				4. **票据授权服务** TGS
					- 减少密码使用频率，保护用户密码
					- 减轻 AS 负担，提升整个身份认证系统效率
				5. **时间戳** Timestamp
					- 防止票据与认证符 Authenticator 的**重放攻击**
					- 以会话密钥作为认证有效性的依据
			4. v4与v5的差异
				- **加密算法**：v4仅支持DES，v5支持多种
				- **网络协议**：v4仅支持IP，v5支持多地址
				- **报文序列编码**：v4使用自定义编码Receiver-makes-right，v5使用标准 ASN.1 BER
				- **票据生命周期**：v4最长1280分钟，v5无限制
				- **支持认证传递**：v5支持，v4不支持
				- **跨域认证**：v4：跨域复杂，密钥交换多，v5支持跨域认证，减少密钥交换
			5. 弱点
				1. **时间同步问题**
					- 时间同步难以实现
					- 攻击者可篡改系统时间，实施重放攻击
				2. **猜测攻击**（弱口令）
					- 弱密码易被攻击
					- 协议本身对密码无额外保护
				3. **密钥存储问题**
					- 密码与会话密钥无法安全存储在普通计算机系统中。
2. **授权与访问控制** Authorization & Access Control
	1. 概念
		- **核心问题**：验证你被允许执行所尝试的操作，是**身份认证后的深度权限控制**
		- **三者等价**：**授权=访问控制**，核心是管理主体对客体的操作权限
		- **三要素**：
			- **主体** Subject：访问者，用户/进程
			- **客体** Object：被访问者，文件/数据/程序
			- **权限** Privilege：操作，读/写/删/执行/授权
		- **安全访问控制核心**：
			- **三大功能**：**授权** Authorization、**撤销** Revoke、**检查** Checker
			- **两个阶段**：**制定策略** Make Policy、**执行策略** Execute Policy
	2. **三大访问控制模型**
		1. **自主访问控制** DAC  Discretionary Access Control
			- **特征**：根据主体身份和授权决策，自主决定意味着**拥有某些特权的主体**能够**自动**将其部分访问权限**授予其他主体**
			- **缺点**：**权限控制不严格**，易因权限传递导致越权访问
			- **实现方式**：
				- **访问控制矩阵** Access Control Matrix：行主体、列客体，单元格为权限
				- **访问控制列表** ACL：以客体为中心，记录可访问它的主体列表，防止访问控制矩阵的稀疏
				- **能力列表** Capability List：以主体为中心，记录它可访问的客体列表
			- **典型示例**：Unix系统文件权限
				1. 分**三类用户**：文件所有者 u、同组用户 g、其他用户 o
				2. 分**三种权限**：读r写w执行x，用9位二进制/八进制表示，如rwx=7、r--=4
				3. 示例：![Screenshot_20260401_103342](/blog/obsidian-assets/screenshot-20260401-103342-2e091864fd.jpg)
			- DAC **的木马攻击风险**
				- **权限设定**
					- 用户A：对文件F有读写权限 RW，对文件G有写权限 W
					- 用户B：对文件F无读权限，对文件G有读权限 R
				- **攻击流程**
					1. 用户B在正常程序中植入木马
					2. 用户A执行该带木马的程序
					3. 木马**将文件F的内容复制到文件G**
					4. 用户B读取文件G，窃取原本无权访问的F的内容
					![Screenshot_20260401_104940](/blog/obsidian-assets/screenshot-20260401-104940-b6481dff1a.jpg)
				- **缺陷**：DAC 允许主体自主传递权限，**无法防范权限滥用/越权传递**，木马可借合法主体权限完成信息泄露
		2. **强制访问控制** MAC Mandatory Access Control
			- **特征**：基于主体和客体的安全级别强制分配权限，**用户无法自主修改**，由密级决定![Screenshot_20260401_103534](/blog/obsidian-assets/screenshot-20260401-103534-699d499530.jpg)
			- **经典模型**
				1. **Bell-LaPadula 模型**：**保密性模型**
					- 目标：**保障数据保密性**，防止高密级信息泄露给低密级主体
					- 规则：保证信息单向流动，高密级信息不向低密级流露
						1. **简单安全属性** No Read Up，**不上读**：主体只能读取同级别或更低安全级别的客体
						2. **\* 属性** No Write Down，**不下写**：主体只能向**同级别或更高安全级别**的客体写入
				2. **Biba 模型**：**完整性模型**
					- 目标：**保障数据完整性**，防止低完整性信息污染高完整性数据
					- 规则：与 Bell-LaPadula **完全相反**
						1. No Read Down **不下读**：主体不能读取完整性低于自身的客体
						2. No Write Up **不上写**：主体不能向完整性高于自身的客体写入
			- MAC **对木马的防护**
				1. 安全标签设定
					- 主体安全级别
						- 用户A：Secret 密级
						- 用户B：Unclassified 非密级
					- 客体安全级别
						- 文件F：Secret 密级
						- 文件G：Unclassified 非密级
				2. **防护原理**：基于 Bell-LaPadula 模型
					1. 木马以用户A权限运行，可读取文件F
					2. 但木马无法向文件G写入，违反不下写规则
					3. 即使木马被执行，也**无法完成高密级信息向低密级客体的泄露**，从根源阻断攻击
					![Screenshot_20260401_105009](/blog/obsidian-assets/screenshot-20260401-105009-7f416e1cf2.jpg)
				3. 优势：MAC 基于系统强制的安全标签，主体无法自主修改权限，彻底防范木马借权泄露
			- **隐蔽信道** Covert Channel
				1. **定义**
					- B.W. Lampson 定义：不被正常访问控制机制允许、**通过系统资源进行通信**的信道
					- 维基百科定义：一种安全攻击手段，可在**原本被安全策略禁止通信的进程间**，建立信息传输能力
				2. **特点**
					- 不依赖正常数据通路，**利用系统资源**，如CPU占用、磁盘空间、响应时间传递信息
					- **MAC** 可阻断显式信息泄露，**但难以完全防范隐蔽信道**
					- 是高安全等级系统需要重点防护的攻击面
				3. **常见类型**
					- **存储隐蔽信道**：利用共享存储资源，如文件大小、磁盘剩余空间传递信息
						- **资源耗尽型隐通道**：高安全级进程与低安全级进程通过共享内存池通信，约定 “申请耗尽内存代表比特 1，不申请代表比特 0”，低安全级进程通过内存申请结果即可接收高安全级进程传递的信息，完全绕过 MAC 的 “不下写” 规则
					- **时间隐蔽信道**：利用系统响应时间、CPU调度等时间相关资源传递信息
						- **负载感知型隐通道**：高安全级进程通过 CPU 占用率传递信息，约定 “占满 CPU 代表比特 1，休眠释放 CPU 代表比特 0”，低安全级进程通过自身计算任务的耗时，即可还原传输的比特信息
				4. **应对策略**：
					1. **前提**：隐蔽信道无法被彻底消除，只能通过手段降低其存在风险与带宽
					2. **通用应对手段**
						1. **关闭或减缓信道**：直接切断非授权资源访问路径，或通过调度、限流降低资源传输速率
						2. **检测滥用行为**：异常行为监控、审计日志分析
						3. **容忍存在**：将泄露风险控制在可接受范围
				 5. 具体防御方法
					1. **限制共享资源**
						- 仅允许相同安全级别主体共享系统资源
						- 隔离跨级别资源访问，切断高/低密级进程间的间接通信链路
					2. **限制信道带宽**
						- 引入干扰disturbing或噪声noise，破坏通信规律
						- 增加资源访问延迟、随机化资源分配
		3. **基于角色的访问控制** RBAC
			1. 产生背景
				- 解决**DAC安全性弱**、**MAC灵活性差**的问题，减少安全管理员的权限管理工作量，源于操作系统的组概念
				- **思想**：**将权限分配给角色**，再**将角色分配给用户**，用户通过角色获得权限
			2. 要素与概念
				- 五大基本元素：**用户** USERS、**角色** ROLES、**客体** OBS、**操作** OPS、**权限** PRMS
				- 角色与组的区别：
					- **组** Group=**用户**集合
					- **角色**=**用户**集合+**权限**集合
				- **关系**：用户与角色、角色与权限均为**多对多映射**
				- **会话** Session：用户与激活角色的映射
			3. 三大扩展模型：
				- **核心RBAC**：基础用户-角色-权限关联，用户通过会话激活角色，仅使用当前角色权限，实现最小权限![Screenshot_20260401_111941](/blog/obsidian-assets/screenshot-20260401-111941-357b81950c.jpg)
				- **层次RBAC**：支持角色继承，符合组织层级
					- 继承语义：子角色拥有父角色的全部权限，并可新增权限
					- 关系特性：自反性、传递性、反对称性
					- 示意图：![Screenshot_20260401_112118](/blog/obsidian-assets/screenshot-20260401-112118-acf838194a.jpg)
				- **约束RBAC**：增加职责分离，防止权限越界
					- **静态职责分离** SSD Static Separation of Duty
						- 约束**用户—角色分配**阶段
						- 规则：一个用户**不能同时拥有互斥角色**
						- 作用：从根源避免一人掌握全套违规操作权限
					- **动态职责分离** DSD Dynamic Separation of Duty
						- 约束**会话—角色激活**阶段
						- 规则：同一会话中**不能同时激活互斥角色**
						- 作用：运行时强制最小权限，防止临时越权
			1. 特性
				- **职责分离**：静态职责分离，SSD，用户不可同时分配互斥角色，如管理员与审计员
				- **角色层级与继承**
				- **角色动态激活**
				- 用户/角色成员**关系约束**
			2. 优势
				- 授权管理简单高效
				- 易实现最小权限原则
				- 适配企业业务与组织架构
				- 支持任务分工与分级管理
				- 可兼容实现DAC与MAC
		4. **安全访问控制原则**
			1. 模型固定，策略灵活：合适的安全策略是安全的关键
			2. **最小特权原则**：将超级用户权限**拆分给系统管理员、安全管理员、审计管理员**，三者相互制约
			3. 授权管理方式
				- DAC：**集中管理**、层级管理、所有者管理、协同管理、分布式管理
				- MAC：**仅由系统管理员分配**主体/客体安全级别，客体级别继承创建者级别
				- RBAC：**支持自主管理**，权限管理委托为核心特征，DAC/MAC无此特性
# 四、网络与安全
## 1. IP 安全
1. TCP/IP 协议栈
	1. **网络结构**：互联网由骨干网、本地网络、ISP组成
	    - 使用 **TCP/IP 协议栈**进行**路由和连接**
	    - 使用 **BGP** **边界网关协议** 进行**路由器发现**
	    - 使用 **DNS** **域名系统** **查找 IP 地址**
	    - ![Screenshot_20260408_101254](/blog/obsidian-assets/screenshot-20260408-101254-fc83686d90.jpg)
	2. TCP 协议栈
		1. 结构示意图：![Pasted image 20260401113404](/blog/obsidian-assets/pasted-image-20260401113404-8c018fb47b.png)
		2. 分层结构
			1. **应用层** Application Layer
				- 对应 OSI 上三层 应用/表示/会话
				- **核心协议**：HTTP/HTTPS、FTP、SMTP、DNS、SSH、TLS
				- 功能：应用业务逻辑、数据格式化、加密、会话管理
			2. **传输层** Transport Layer
				- **核心协议**：**TCP** 可靠、**UDP** 不可靠
				- 功能：端到端进程通信、端口号区分服务、可靠性控制
				- **TCP 与 UDP 区别**：TCP面向连接、可靠字节流服务，发送后检查是否成功，若失败继续发送直到成功；UDP不连接，只负责发送，不保证成功
			3. **网际层** Internet Layer
				- 核心协议：IP IPv4/IPv6、ICMP、ARP、IGMP
				- 功能：寻址、路由、分组转发、IP地址管理
			4. **网络接口层** Link Layer
				- 对应 OSI 数据链路层+物理层
				- 协议：以太网、Wi‑Fi、PPP、MAC
				- 功能：帧封装、物理传输、MAC寻址、差错检测
	3. **数据封装格式**
		- **应用层消息** message→**传输层**切分后，添加TCP/UDP头部成为**段** segment→**网络层**添加IP头部成为**包** packet→**数据链路层**添加以太网头和尾成为**帧** frame
		- **传输层**切分数据，把虚拟流量**分到不同的端口**上，**并行加速处理**
		- **网络层**添加IP头，得到**发送目的地**
		- **数据链路层**添加以太网头和尾，传输的过程中，**不同的网络会添加不同的头尾**，并且添加之前会**去掉之前添加的头尾**，用于适配当前的网络接口要求
		- ![Screenshot_20260408_101407](/blog/obsidian-assets/screenshot-20260408-101407-eaf55a5d14.jpg)
	4. **协议特性**
		- **IP** Internet Protocol：**无连接、尽力交付**的**不可靠协议**，通过**数字地址路由**，路由中通常包含若干**跳**hop，报头含协议字段（TCP=6、UDP=17）
			- **子网掩码** Subnet Mask：一个32位二进制数，和IP地址配合使用，子网掩码里是 **1** 的位 → 对应IP的**网络部分**，子网掩码里是 **0** 的位 → 对应IP的**主机部分**
			- **路由表**：路由器的导航地图，主要包含这几列：
				- 目的网络 / 目标网段
				- 子网掩码
				- 下一跳 Next Hop：数据包下一步交给哪个设备
				- 出接口
				- 度量值 / 优先级
			- **网关**：同一网段设备，直接通信，不用网关；不同网段设备，必须走网关
			- **IP头部结构**
				- **关键字段**：
					- 8-bit Time To Live  **TTL**：**生存时间**，每经过一个路由器减1，**防止环路**
					- 8-bit Protocol：**协议字段**，标识上层传输层协议
				- ![Screenshot_20260408_105715](/blog/obsidian-assets/screenshot-20260408-105715-558665b5a8.jpg)
			- **协议字段** Protocol Field
				- **作用**：IP包通过该字段区分上层传输层协议，实现数据分用
				- **协议号**：
					- **TCP**：协议号6，面向连接 Connection-oriented
					- **UDP**：协议号17，无连接 Connectionless
				- 层级关系：IP层 Internet Layer 为**无连接协议**，为上层TCP/UDP提供路由转发服务
				- ![Screenshot_20260408_110056](/blog/obsidian-assets/screenshot-20260408-110056-b5639e99f8.jpg)
			- 发送示例![Screenshot_20260408_102726](/blog/obsidian-assets/screenshot-20260408-102726-c2857de774.jpg)
		- **ICMP**：承载在IP包中，提供**网络操作反馈**，如错误报告、可达性测试等
		- **TCP** Transmission Control Protocol：面向连接、可靠
			- 通过以下方式建立连接
				- **滑动窗口** sliding windows：流量控制
				- **序列号+确认号** sequencing numbers and acknowledgments：可靠性
				- **三次握手同步** synchronization：通信双方同步彼此的初始TCP序列号**ISN**，建立连接
			- **工作机制**
				- **发送端** Sender：将完整数据拆分为多个分段 Segments，为每个分段添加序列号 Sequence Number
				- **接收端** Receiver：根据序列号重组分段，恢复原始数据；发送确认 Acknowledgment，**对丢失的分段请求重传**
				- **连接状态维护**：通信双方全程维护连接状态，保障传输可靠性
			- **滑动窗口**
				- **发送窗口**：发送方允许发送的数据范围，包含已发送未确认、未发送但可发送的数据
				- **接收窗口**：接收方允许接收的数据范围，保证按序接收并缓存待处理数据
				- **工作流程**
					1. **初始化**：连接建立时，双方根据初始缓冲区大小确定窗口大小
					2. **发送方行为**：
						- 仅发送窗口内的数据，超出窗口范围的数据需等待窗口滑动
						- 收到ACK确认后，窗口左边界右移，释放已确认的缓冲区空间
					3. **接收方行为**：
						- 接收数据后，将有效数据放入窗口对应位置，更新字段反馈给发送方
						- 完成数据处理并按序交付后，窗口右边界右移，扩大可接收范围
					4. **滑动机制**：窗口整体向后移动，实现数据的连续、有序传输，避免发送方速率过快淹没接收方
			- **TCP三次握手** 3-way Handshake
				- 目的：通信双方同步彼此的初始TCP序列号ISN，建立连接
					- **序列号ISN作用**：**跟踪数据包顺序**，确保传输无丢包；初始序列号是TCP连接建立时的起始编号
				- **报文种类**
					- **SYN** Synchronize：同步报文，用来**建立连接、初始化序列号**
					- **ACK** Acknowledge：确认报文，用来**确认对方序列号有效**
				- **握手流程**：
					1. 主机A发送SYN报文 seq = x
					2. 主机B接收SYN，回复SYN+ACK报文 seq = y，ACK = x+1
					3. 主机A接收SYN+ACK，回复ACK报文 ACK = y+1
					![Screenshot_20260408_111249](/blog/obsidian-assets/screenshot-20260408-111249-3681a95b7c.jpg)
				- **连接建立后交换初始序列号**，保障所有丢失数据可恢复
			- **端口号** Port Numbers
				- **定义**：应用开发者遵循RFC 1700约定的知名端口号，用于**标识应用服务**
				- 端口号位于**TCP头部起始位置**，用于区分同一主机上的不同应用进程
				- ![Screenshot_20260408_111533](/blog/obsidian-assets/screenshot-20260408-111533-ec61bdc163.jpg)
			- TCP头部结构![Screenshot_20260408_111620](/blog/obsidian-assets/screenshot-20260408-111620-96af7dfc76.jpg)
		- **UDP**：无连接、轻量，无确认/流量控制，不保证消息连续、不乱序重排，**通过端口区分应用**，仅完成基本数据传输
			- **目的端口号**：将 UDP 数据报交给对应应用程序
			- **源端口号**：用于对方回复时的返回地址
		- **DNS** Domain Name Service：域名与IP映射，采用**递归/迭代解析**，从本地服务器逐级查询至根服务器
			- ![Screenshot_20260408_111729](/blog/obsidian-assets/screenshot-20260408-111729-001df7cd3e.jpg)
2. **TCP/IP协议栈安全问题**
	1. **嗅探** Sniffing
		- 问题：FTP/Telnet/HTTP 等协议**明文传输数据**，网卡混杂模式可读取所有网络数据
		- 工具：Tcpdump/libpcap、Wireshark、Commview
	2. **ARP欺骗/中毒** ARP Spoofing/Poisoning
		- 原理：ARP是无状态协议，部分系统仅接受首次ARP响应，攻击者伪造ARP包**篡改目标ARP缓存**
			- ARP：IP → MAC 地址解析
			- RARP：MAC → IP 反向地址解析
		- 危害：可发起拦截、中间人、拒绝服务攻击
	3. **IP欺骗** IP Spoofing
		- **原理**：伪造IP源地址发送请求，典型为Smurf攻击，**伪造受害者IP向广播地址发ICMP请求**，引发大量回复拥塞受害者
		- Smurf攻击：![share_1775619001195](/blog/obsidian-assets/share-1775619001195-d5c30cf2c7.png)
	4. **TCP SYN泛洪** TCP SYN Flooding
		- **原理**：攻击者**伪造IP发送大量SYN请求**，服务器为每个请求分配资源（线程、内存），但**一直无法收到回复**的$ACK_s$，因此会一直开线程直至耗尽，无法处理正常连接![Screenshot_20260408_113544](/blog/obsidian-assets/screenshot-20260408-113544-cad16e331e.jpg)
		- **防护**：
			- **随机删除** random delete：如果满队列，随机删除一半连接，**方便修改**，不需要修改客户端，**直接修改服务器**即可
			- **SYN Cookies**：如果全部修改，需要修改客户端，在协议修改上难以实现，因此在三次握手的基础上略微修改
				- 服务器**不会保存任何连接状态**，除非它收到来自客户端的**至少两条报文**
				- 服务器会将套接字信息，即**服务器与客户端的 IP 地址及端口号**编码为一个 Cookie，并将该 Cookie 发送给客户端
				- 客户端必须**在第二次请求中带上这个 Cookie**，服务器会重新计算 Cookie 并与客户端发来的 Cookie 进行**比对校验**
				- **SYN Cookie 的实现**：Linux/FreeBSD原生支持，在旧版 Linux 系统中，SYN-ACK Cookie 功能默认未开启。系统管理员需要在启动脚本中添加 `echo 1 > /proc/sys/net/ipv4/tcp_syncookies` 来启用
				![Screenshot_20260413_135343](/blog/obsidian-assets/screenshot-20260413-135343-f7747d9c40.jpg)
		- 属于典型的 **DoS** **拒绝服务攻击**：**DoS攻击特征**
			- 攻击发起方几乎不消耗自身资源，而被攻击方却需要为每一个请求单独创建线程进行处理
			- **存在非对称性**：攻击成本极低，受害方应对成本极高
			- **DDoS** **分布式拒绝服务攻击**：攻击者控制成千上万台被入侵的傀儡机（肉鸡），从不同地域、不同网络同时发起攻击，形成超大流量洪峰
	5. **TCP SYN预测攻击**
		- **原理**：TCP 协议中的每个数据包都拥有一个**序列号** ISN Sequence Number，接收方会依据该序列号对数据包进行排序与重组。若**攻击者预测出序列号**，便可向接收方**发送伪造数据包**，使其按照攻击者的意图重组报文，达成攻击目的
		- TCP 序列号预测攻击是诸多其他攻击的基础，包括：
			- TCP 欺骗 spoofing
			- TCP 连接劫持 connection hijacking
			- TCP 重置 reset
		- **防护**：采用**随机ISN**，RFC6528建议ISN结合定时器和伪随机哈希函数生成
	6. **TCP拥塞控制攻击** Congestion Control
		- **背景**：当发生数据包丢失时，**说明网络出现拥塞**，TCP 协议要求发送方执行以下操作：
			- **将传输速率减半**，并持续减半，**直到不再丢包或速率降至 0**
			- 若**丢包停止**，则再**缓慢提高**传输速率
		- **原理**：恶意节点违反拥塞控制协议，在网络丢包时不降低速率，抢占合法用户带宽
		- **防护**：添加ACK随机数，验证ACK合法性防止作弊
	7. **DNS欺骗/缓存投毒** DNS Spoofing/Cache Poisoning Attack
		- **原理**：篡改DNS服务器/本地缓存，将域名解析至错误IP，引导流量至攻击者服务器
		- DNS缓存投毒为何会发生：
			- DNS请求与响应未进行身份认证
			- 攻击者可以提供伪造的DNS信息
		- **DNSSEC** **域名系统安全扩展**：**对DNS响应做认证**，**无加密/抗DoS能力**，定义于RFC4033/4034/4035
3. **IP层安全** **IPSec**
	1. **基础概述**
		- **作用**：为IP层所有流量提供加密和认证，IPv6必须支持 IPSEC（君子协议）、IPv4可选支持
		- **核心标准**：
			- RFC2401 安全架构 Security Architecture
			- RFC2402 AH
			- RFC2406 ESP
			- RFC2408 ISAKMP Internet Security Association and Key Management Protocol **Internet 安全联盟和密钥管理协议**
		- **三大核心组件**：认证头 AH IP Authentication Header、封装安全载荷 ESP Encapsulating Security Payloads、安全关联 SA Security Associations
	2. **组件**
		1. **AH 认证头**：为IP数据包提供**数据完整性和身份认证**服务，可选**抗重放**保护，无加密功能，即**只验证，不加密**
			- ![Screenshot_20260413_142747](/blog/obsidian-assets/screenshot-20260413-142747-23c13eb8d9.jpg)
			- **下一个头部** Next Header，8位：标识紧跟在AH之后的**下一个头部类型**
			- **有效载荷长度** Payload Len，8位：以32位字为单位的**AH长度**，再减2
			- **保留位** Reserved，16位：备用字段
			- **安全参数索引** SPI Security Parameters Index，32位：与该IP数据包相关的安全联盟SA索引
			- **序列号** Sequence Number，32位：单调递增的计数器，**用于防止重放攻击**
			- **完整性校验值** ICV：包含消息认证码MAC或完整性校验值
		2. **ESP 封装安全载荷**：**提供机密性，加密**、可选认证/抗重放，报头含SPI、序列号，尾部含填充信息、ICV，**验证＋加密**![Screenshot_20260413_143125](/blog/obsidian-assets/screenshot-20260413-143125-dbbbdfed77.jpg)
		3. **SA 安全关联**：单向的加密/认证算法+参数（密钥）集合，双向通信需要2个SA；由SPI、目的IP、安全协议AH/ESP唯一标识，通过ISAKMP协商建立，简而言之，**SA是一组逻辑上的安全参数集合**，用于实现与另一实体间的安全信息交互
			- 三个参数可唯一确定一个SA：
				- **安全参数索引** SPI：与该SA关联的位串
				- **IP目的地址**：该SA的目的地址
				- **安全协议标识符**：指明使用AH协议还是ESP协议
			- SA 通过 RFC2408 中定义的**互联网安全联盟与密钥管理协议** ISAKMP 进行密钥交换
				- 该协议提供**建立安全关联的机制**，并定义**密钥交换框架**，但其本身**不包含具体的密钥交换算法**
				- 常用的密钥交换协议包括：
					- **互联网密钥交换 IKE**：RFC2409 是应用最广泛的版本
					- **Oakley 密钥确定协议**：基于优化后的迪菲-赫尔曼 Diffie-Hellman 算法
					- Kerberized Internet Negotiation of Keys **KINK**，**Kerberos 互联网密钥协商**:定义于 RFC4430，基于 Kerberos 协议实现，采用对称密钥技术
	3. **工作模式**
		1. **传输模式** Transport Mode：**仅保护IP包载荷**（**传输+应用层**），不修改IP头，不影响路由，适用于主机间点对点通信![Pasted image 20260413144010](/blog/obsidian-assets/pasted-image-20260413144010-6660892da7.png)
		2. **隧道模式** Tunnel Mode：**加密/认证整个原始IP包**，封装为新IP包，适用于VPN、网络间通信![Pasted image 20260413143926](/blog/obsidian-assets/pasted-image-20260413143926-8726707d84.png)
	4. **算法与优势**
		- 算法：![Screenshot_20260413_144300](/blog/obsidian-assets/screenshot-20260413-144300-c3c3b6c6ad.jpg)
		- **推荐算法**：ESP加密用AES-CBC-128，AH/ESP认证用HMAC-SHA1-96
		- **优势**：
			- IPSec **可在防火墙/路由器中实现并强制启用**
				- 所有经过网络边界的数据包都会被增强安全防护
				- 受防火墙保护的主机无需自行处理安全问题
			- IPSec **对终端用户透明**(无感)
				- 基于 IPSec 网络开发的应用程序无需做任何特殊修改
				- 保密性与完整性会自动得到保障
				- 但控制粒度较低，不易修改
4. **TCP层安全**：SSL/TLS协议栈
	1. **发展历史**
		- **SSL** Secure Socket Layer **安全套接层** 由网景Netscape提出，**TLS** Transport Layer Security **传输层安全协议** 为其标准化升级
		- **核心版本**：
			- TLS1.0：SSL3.1，RFC2246
			- TLS1.1：RFC4346
			- TLS1.2：RFC5246
		- **作用**：SSL/TLS 协议的核心是为 TCP 连接提供端到端的**保密性、完整性、身份认证服务**。**可信根 CA** 签发的 SSL 证书，唯一作用是**证明域名的合法归属、校验服务器的真实身份**，确保客户端连接的是域名对应的合法服务器，而非伪造服务器，其生效环节是**TCP 连接建立后的 SSL/TLS 握手阶段**
	2. **概念**
		- **SSL连接** Connection：连接是一种提供对应服务类型的传输逻辑（OSI 模型层面的定义），具体是**点对点的临时传输关系**，关联一个SSL会话
		- **SSL会话** Session：**客户端与服务器的长期关联**，由握手协议建立，共享一组连接所共享的密码安全参数，避免重复协商开销
	3. **SSL/TLS** **协议栈结构** Protocol Stack
		位于**传输层与应用层之间**，分两层：
		1. **握手层** Handshake Layer：含握手、更改密码规范、告警子协议，负责身份认证、算法/密钥协商
		2. **记录层** Record Layer：对应用层数据进行分片、压缩（可选）、计算MAC、加密，添加记录头后传至TCP层
		![Screenshot_20260413_145603](/blog/obsidian-assets/screenshot-20260413-145603-a5748b22cf.jpg)
	4. 协议流程
		1. **握手协议** 四阶段：建立安全协商→服务器认证与密钥交换→客户端认证与密钥交换→完成密码规范切换
			![Screenshot_20260413_145651](/blog/obsidian-assets/screenshot-20260413-145651-3a3020e2f4.jpg)
		2. **记录协议** 五步：分片→可选压缩→计算MAC→加密→添加SSL记录头（内容类型、版本、压缩长度）
			![Pasted image 20260413150018](/blog/obsidian-assets/pasted-image-20260413150018-e1be8979d9.png)![Screenshot_20260413_145925](/blog/obsidian-assets/screenshot-20260413-145925-596988104a.jpg)
5. **安全防护总结**
	1. TCP/IP**安全问题**
		集中于协议设计的**无状态/明文/弱认证特性**，易被利用进行欺骗、泛洪、嗅探等攻击
	2. **分层安全防护方案**
		- **网络层** IP：**IPSec** AH+ESP+SA，传输/隧道模式，实现底层流量加密认证
		- **传输层** TCP：**SSL/TLS**，实现端到端应用流量安全，适用于HTTP等应用层协议
	3. **关键防护手段**
		- **抗欺骗**：DNSSEC、随机IP/ISN、ARP缓存防护
		- **抗泛洪**：SYN Cookies、资源限制
		- **防嗅探**：协议加密 IPSec/SSL/TLS、避免明文传输
		- **保完整**：MAC/ICV校验、哈希算法认证
## 2.恶意代码
1. **恶意代码定义**
	- **恶意逻辑**：会导致**站点安全策略被违反**的指令集合，核心是突破安全策略、执行非授权操作
	- **分类**：木马 Trojan、病毒 Virus、蠕虫 Worm，扩展类型包括Rootkit、兔子/细菌、逻辑炸弹等
2. 三大类型
	- **木马** Trojan Horse
		1. **定义**：具备用户**可知的显式** overt **用途**，同时**隐藏用户未知的隐蔽** covert **用途**的**独立可执行**程序，**不具备自我复制**能力
		2. 命名来源：1974年Anderson报告中由Dan Edwards命名
		3. 典型案例：
			- **伪装成ls命令的UNIX脚本**：显式功能是列出目录文件 `ls`，隐蔽功能是创建setuid权限的shell![Screenshot_20260415_101343](/blog/obsidian-assets/screenshot-20260415-101343-7fc982f0ff.jpg)
			- **NetBus**：针对Windows NT系统，伪装成游戏程序，作为服务端接受远程控制，可拦截键盘鼠标操作、上传下载文件
			- **可复制木马** Replicating Trojan Horse：也叫**传播型木马** propagating Trojan horse，**能自我复制**，极难检测，典型为**Thompson编译器木马**，1980年实现，**通过修改编译器植入登录后门**，即使源码审计也无法发现
				1. 修改编译器，使其在编译登录程序login时，让login既接受用户的正确密码，**也接受一个固定万能密码**，所有用户通用![Screenshot_20260415_102705](/blog/obsidian-assets/screenshot-20260415-102705-e90995159a.jpg)
				2. 然后再次修改编译器，使其**在编译新版本的编译器自身时**，会**自动把第一步的后门代码插入进去**![Screenshot_20260415_103311](/blog/obsidian-assets/screenshot-20260415-103311-56c7998ae3.jpg)
				3. 重新编译编译器
				4. **删除包含后门修改的源码**，把未被篡改的原始源码放回原处
			- **结论**：**仅靠源码级校验无法防护**不可信代码
		4. **木马隐匿技术**：Rootkit
			- **特征**：**隐身** Stealth，是一组木马程序的二进制可执行代码集合
			- **典型感染路径**：
				1. **密码窃取/字典攻击**获取权限
				2. 利用 rdist、sendmail、loadmodule、rpc.ypupdated、lpr、passwd 等程序的**缓冲区溢出漏洞获取 root 管理员权限**
				3. FTP **下载 Rootkit**，解压、编译并安装
			- **隐匿手段**：
				- **创建隐藏目录**，目录名常使用不可显示字符，肉眼与普通命令难以察觉
				- **替换系统命令**，netstat/ps/ls/du/login等
				- **伪造程序校验和**，使标准命令无法检测攻击者进程、文件、网络连接
			- **发展里程碑**：
				- 1980年代随UNIX普及流行
				- 1999年首个Windows恶意Rootkit NTRootkit 出现
				- 2005年索尼BMG DRM后门事件，其版权保护软件秘密安装Rootkit引发严重安全问题
		5. 木马/Rootkit的**检测与清除**
			- **检测手段**：
				- **可信介质检测**：如干净U盘，光盘
				- 基于**行为/特征码/差异对比**的检测
				- **完整性校验**
				- **内存转储**分析
			- **清除方式**：
				- 普通用户**手动清除极难**，普遍使用安全厂商的工具
				- 部分安全软件通过 Windows 文件系统 API 访问文件，对付不了 Rootkit，要安全查杀，必须直接访问原始文件系统
				- 部分人声称，运行在**已感染OS中的杀毒软件清除不可靠**
				- 个人PC可使用专业安全套件防护
				- **最安全的清除方式**：**重装操作系统**
	- **计算机病毒** Computer Virus
		1. **定义**：将自身插入一个或多个文件中并执行特定操作的程序，**必须具备插入感染阶段**，执行阶段可为空；非独立程序，**依赖宿主代码存在**，**通过感染其他程序传播**
		2. **执行伪代码**：先判断能不能传播->找没中毒的文件->把自己复制进去，并修改程序入口，让它执行新增病毒->执行恶意行为->运行原来的正常程序![Screenshot_20260415_110135](/blog/obsidian-assets/screenshot-20260415-110135-aae283579c.jpg)
		3. **发展里程碑**：
			- 1980年：Apple II出现实验性病毒程序
			- 1983年由研究生Fred Cohen描述，其导师Adleman命名为计算机病毒
			- 1984年实验验证，UNIX系统中最快5分钟即可获取超级用户权限
			- 1986年：Brain（巴基斯坦）病毒，首个IBM PC病毒，感染软盘引导扇区
			- 1987年：MacMag Peace病毒（Macintosh）、Duff的UNIX shell脚本病毒（8天传播46台系统）
			- 1989年：Highland的Lotus 1-2-3宏病毒
		4. **病毒的分类**
			- **引导扇区感染型** Boot Sector Infectors：**插入磁盘引导扇区**，系统开机/读取磁盘时执行，典型为Brain病毒
			- **可执行文件感染型** Executable Infectors：感染.EXE/.COM等可执行程序，可通过**覆盖、前置、环绕、分散插入、伴生**等方式附着，典型为耶路撒冷病毒
				- ![Screenshot_20260415_110527](/blog/obsidian-assets/screenshot-20260415-110527-dc5b90e7cc.jpg)
				- **覆盖式** overwriting：病毒直接覆盖程序的部分内容
					- 病毒直接覆盖写入可执行文件
					- 实现最简单
					- 由于原程序被破坏，很容易被发现
					- ![Screenshot_20260415_110927](/blog/obsidian-assets/screenshot-20260415-110927-e85711a188.jpg)
				- **头部附着/前置** at the beginning：病毒附加在程序开头
					- 隐蔽性更强，因为原程序保持完整
					- 若原程序体积大，复制可能较慢
					- 多次感染会让文件体积持续变大
					- ![Screenshot_20260415_111031](/blog/obsidian-assets/screenshot-20260415-111031-f8c3cb801c.jpg)
				- **环绕** beginning and end：病毒包裹程序
					- 兼具头部/尾部附着型病毒的全部特性
					- 具备清理痕迹、规避检测的能力
					- 示例：附着在生成带文件大小的文件列表的程序中，在程序执行结束后再进行篡改
					- ![Screenshot_20260415_111256](/blog/obsidian-assets/screenshot-20260415-111256-0e7d63d360.jpg)
				- **分散插入** intersperse：利用jump在程序中跳转，在整个文件中分布![Screenshot_20260415_111417](/blog/obsidian-assets/screenshot-20260415-111417-cfc9be4984.jpg)
				- **伴生** companions：![Screenshot_20260415_111456](/blog/obsidian-assets/screenshot-20260415-111456-0061835ee4.jpg)
			- **多分区复合型** Multipartite viruses：可同时感染引导扇区和可执行文件
			- **TSR驻留型** TSR viruses：程序执行后驻留内存保持活跃，典型为Brain、耶路撒冷病毒
			- **隐身型** Stealth viruses：通过修改系统中断处理程序，隐藏文件感染特征，如返回未感染文件大小、临时消毒后再感染
			- **加密型** Encrypted viruses：除解密程序外，**病毒主体全部加密**，使用密钥/加入无用代码等，大幅提升特征检测难度
			- **多态型** Polymorphic viruses：**每次复制时改变自身形态**（指令替换、算法变更），规避特征码检测，两个代码段的语义等价性判定是不可解问题![Screenshot_20260415_112906](/blog/obsidian-assets/screenshot-20260415-112906-71bfc725d0.jpg)
			- **宏病毒** Macro viruse：存储在文档/表格的命令集中，文件打开时加载执行，可跨平台传播
	- **蠕虫** Worm
		1. **定义**：**独立的可执行程序**，可通过自我复制自动向目标系统传播，无需依赖宿主文件
		2. **与木马、病毒的差异**：存在形式 与 自我复制/传播能力
			- **病毒**：插入宿主代码，**非独立**程序、**感染其他程序**实现传播
			- **木马**：**独立**可执行程序、**无自我复制**能力
			- **蠕虫**：**独立**可执行程序、**自动自我复制**，**跨系统传播**
		3. 起源：1982年Schoch & Hupp的分布式计算研究
		4. 典型案例：
			- 1987年：**圣诞蠕虫**，IBM网络专用，绘制圣诞树并发送祝福，同时读取通讯录批量传播，导致多个IBM网络瘫痪
			- 1988年：**Morris蠕虫**，首个大规模互联网蠕虫，原本用于测量互联网规模，因代码bug无限复制导致感染主机崩溃，造成1000万-1亿美元损失；其作者是首个依据《计算机欺诈和滥用法》被起诉的人
			- 2001年：**Code Red系列**，现代意义首个蠕虫，利用IIS服务器缓冲区溢出漏洞，14小时感染35万台主机，携带DDoS攻击载荷；Code Red II优化扫描策略，植入后门实现无限制远程访问
			- 2001年：**Nimda**，多模式传播蠕虫，可通过IIS漏洞、邮件附件、共享文件夹、网页恶意代码传播，催生零日攻击安全问题（特征码更新前就完成大规模感染）
			- 2003年：**Slammer**，UDP蠕虫，利用SQL Server缓冲区溢出漏洞，完整代码仅404字节，8.5秒感染量翻倍，10分钟内饱和互联网带宽，核心传播优势是无连接UDP协议，不受TCP三次握手延迟限制
			- 2003年：**Blaster蠕虫**，利用Windows RPC服务漏洞，发起DoS攻击并植入后门；Welchia/Nachia“善意蠕虫”，本意修复漏洞、清除Blaster，却耗尽网络带宽造成更大破坏
3. 其他类型恶意代码
	- **兔子/细菌** Rabbits, Bacteria
		- **定义**：持续**耗尽系统某类资源的程序**，典型为UNIX下循环创建目录的脚本，可耗尽磁盘空间/inode表导致系统崩溃
		- **防御手段**：针对用户设置资源配额
	- **逻辑炸弹** Logic Bombs
		- **定义**：当**特定外部事件**（如特定日期、特定记录删除）**触发时**，执行违反安全策略操作的程序
		- **典型案例**：程序在开发者薪资记录被删除时，清空公司全部薪资记录
4. **恶意代码防御体系**
	- **理论前提**
		- 1989年Fred Cohen证明：**不存在能够检测所有可能病毒**/恶意代码的通用算法，没有完美的防御方案
		- **病毒检测的矛盾**：新病毒传播速度极快（从早年数月到如今数小时），人工分析提取特征码的速度难以匹配
	- **主流防御技术与手段**
		- **基于特征码的杀毒技术**
			- **原理**：在文件中查找**已知恶意代码的特定字节序列**，即**病毒特征码**，发现后告警/消毒
			- **局限性**：无法应对**未分析的新病毒、多态/变形病毒**，必须持续更新特征库
		- **数据与可执行文件严格隔离**
			- **原理**：病毒必须写入程序才能感染，**数据不允许执行**，**将数据转为可执行文件**必须经过可**审计**的操作
			- **典型案例**：LOCK系统，编译后的程序默认为数据类型，必须经过特定可审计事件才能转为可执行类型，且不可修改可执行对象，阻断病毒感染
		- **最小权限原则落地**
			- **原理**：**剥夺进程不必要的权限**，使其仅能完成预设功能，限制恶意代码篡夺用户权限后的破坏范围
			- 概念：任何主体（用户、进程、程序、服务、设备等）在执行其合法任务时，仅被授予完成该任务**必需的最小权限集合**，且权限的**生效时长仅限任务执行的必要周期**，严禁授予超出任务需求的任何多余权限
		- **访问控制与隔离**
			- **守护程序/看门狗**：拦截所有**文件打开请求**，校验访问权限，重定义系统调用
			- **沙箱/虚拟机**：将程序运行在**受保护的隔离区域**，替换为受限权限的系统调用/库
			- **共享限制**：基于完整性策略做隔离，执行程序前校验用户对程序所有者的**信任关系**
			- **多级安全机制**：将可执行文件置于低于用户和管理区域的“**病毒防护区**”，阻止恶意代码写入
		- **文件变更检测**
			- **原理**：为每个文件计算**操作检测码** MDC 生成签名块，**定期重新计算并比对**，发现文件变更
			- 典型工具：tripwire，签名包含文件属性、多种密码学校验和
		- **异常行为与规范外操作检测**
			- 原理：**将感染视为错误**，执行程序时应用**容错技术**，校验程序执行的合规性
			- 典型方案：
				- 指令序列校验和加密
				- **N版本编程**：多版本算法并行执行，多数表决
				- 携带证明代码：PCC，代码附带满足安全要求的证明，用户校验后才执行
		- **统计特征分析**
			- 原理：基于**编码风格、系统资源使用**的统计特征，检测异常代码与异常行为
			- 局限性：通常在**损害发生后才能检测到**
5. 僵尸网络、DDoS与垃圾邮件
	- **僵尸网络** Botnet
		- **定义**：由大量**可远程受控的僵尸** zombie **主机组成的网络**，主机所有者通常不知情，通过IRC（群聊）或P2P（点对点访问）协议受控，是多种网络攻击的承载平台
		- **感染流程**：Win XP
			- 利用漏洞执行shellcode，如缓冲区溢出、邮件病毒等攻击方式
			- shellcode下载安装bot程序
			- bot关闭防火墙/杀毒软件
			- 连接IRC服务器加入受控频道
				- 通常需要通过 DNS 解析获取服务器 IP 地址，尤其在服务器原始 IP 已被列入黑名单时
				- 认证密码一般直接内置在僵尸程序的二进制文件中
			- Botmaster通过认证下发指令
			![Screenshot_20260420_142511](/blog/obsidian-assets/screenshot-20260420-142511-614f5882d5.jpg)
		- **传播流程**：
			1. 每台僵尸主机Bot均可扫描IP地址空间，寻找新的受害者
				- 自动扫描
				- 受控指令触发：针对特定的 /8 或 /16 网段前缀进行定向扫描
				- 僵尸网络控制者 Botmaster 会**共享网段信息**，避免重复扫描
			2. 僵尸网络的**主动管理**
				- **检测失去响应的僵尸主机**，并**识别出超级僵尸主机** super bots
			3. 僵尸网络之间的战争证据
				- 通过大量IRC连接（克隆连接）对服务器发起拒绝服务攻击DoS
				- N-to-N架构，参考分布式拒绝服务DDoS中的相关架构
		- **用途**：分布式拒绝服务攻击 DDoS、垃圾邮件发送、点击欺诈、新漏洞/蠕虫的传播跳板
		- 典型案例：Conficker僵尸网络，感染超1050万台主机，每日可发送100亿封垃圾邮件；BredoLab僵尸网络，每日垃圾邮件发送量达36亿封
	- **拒绝服务攻击** DoS 与**分布式拒绝服务攻击** DDoS
		- **DoS目标**：**耗尽目标主机资源**，拒绝为合法客户端提供服务
			- 典型DoS攻击类型：Smurf攻击、死亡之Ping、SYN洪泛、UDP洪泛
		- **DDoS原理**：利用僵尸网络的多层架构，通过数千个不同来源的僵尸主机**向目标发起流量攻击**，彻底耗尽目标带宽/资源，**无需地址伪造**
	- **垃圾邮件** SPAM
		- **定义**：**未经接收者同意批量发送的电子邮件**，驱动是**商业利益**，制作与发送门槛极低
		- **传播支撑**：邮件地址爬取工具、批量发送工具、僵尸网络、配置错误的Open Relay邮件服务器
		- **规避手段**：
			- **邮件伪造**：**SMTP无原生认证**，可伪造MAIL FROM字段
			- 利用僵尸网络分散发送IP
			- 劫持未分配IP地址空间
		- **反垃圾邮件技术手段**：
			- **过滤类**：**关键词过滤**、**黑白名单**、**HASH过滤**、**规则过滤**、**贝叶斯过滤**
				- 发送方校验：SMTP认证、反向解析、DKIM、SenderID、邮件指纹
				- 挑战认证：挑战-响应机制、计算挑战
			- **反垃圾邮件法律规范**：
				- 美国：2003年《CAN-SPAM法案》
				- 中国：2006年实施《互联网电子邮件服务管理办法》
